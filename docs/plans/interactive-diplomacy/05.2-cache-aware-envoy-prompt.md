# Cache-aware envoy prompt structure

> Part 2 of the diplomat refactoring. **Depends on `05.1-send-message-tool.md` being shipped first.** It relies on `send-message` tool-call parts existing in history, on the shared `collectSpokenReply` join (05.1 shipped it in `utils/diplomacy/transcript-utils.ts`, re-exported from `transcript.ts`), and on the shared deal-state predicate `counterpartOpenProposal` (05.1 shipped it in `deal-reduce.ts`; this part adds the `ownPendingProposal` companion).
>
> This is a **performance** change (caching + token reduction), independent of 05.1's behavioral fix. Keep it small: a fixed compaction boundary, not a sticky threshold.

## Problem

`LiveEnvoy.getInitialMessages` rebuilds the whole conversation into many small `ModelMessage`s every run, deal rows are reduced to their bare one-line text (terms stripped), and there is no cache breakpoint — so the long stable history is neither compact nor cacheable.

## Goal

Collapse the settled conversation (everything before the active exchange, hydrated from the durable store) into a single, cacheable `# Past Conversation` user block that **includes deal summaries**; the current activity trails it verbatim/additively.

### Shape
```
[system]                                   (getSystem)
[game context messages]                    (buildGameContextMessages — stable)
{ role: 'user', content: "# Past Conversation\n<compacted transcript + deal summaries>",
  providerOptions: { anthropic: { cacheControl: { type: 'ephemeral' } } } }   ← CACHE BREAKPOINT
<live tail: the not-yet-compacted recent messages, emitted verbatim/additively>
{ role: 'user', content: hint + addon }     (turn-specific — uncached)
(Diplomat) { role: 'user', content: dealContext }   (current on-the-table deal — uncached)
```

## Boundary: fixed = hydrated length (kept deliberately simple)

The compaction boundary is **the hydrated length**, recorded once per thread when `syncThreadMessages` rebuilds `thread.messages`. Everything up to the boundary is folded into `# Past Conversation`; everything after is the live tail.

- Stored in a module-level `Map<threadId, number>` (server-only; never on the wire-serialized `EnvoyThread`).
- **Re-hydration advances it** to the new hydrated length (open/refresh re-reads the store and re-bounds), folding the session's just-sent turns from the live tail into the head. So a refresh after new turns is **not** a cache no-op: the head *extends* by exactly those turns and the provider re-caches the now-longer prefix **once**. What stays invariant is the *prefix* — the head only ever grows by appending settled turns, never rewrites earlier ones, and reopening at the **same** durable state reproduces a byte-identical head — so the cache extends cleanly instead of busting (see the byte-stable per-line rendering below).
- Ordinary (non-diplomacy) chats have no hydration → boundary defaults to 0; their whole transcript is the live tail.

> **No sticky threshold.** An earlier draft advanced the boundary only when the tail's token count crossed `COMPACT_THRESHOLD`, to keep the head byte-stable as the tail grows mid-session. That machinery (a token count per step, a threshold constant, advance logic) isn't worth it: within a single server session the live tail is just that session's turns (small), and each reload re-hydrates and re-bounds anyway. Start with the fixed boundary; only add threshold-advancing later if a pathologically long single session ever proves it necessary.

## Reusable compaction (new module `vox-agents/src/envoy/utils/conversation-compaction.ts`)

- `renderPastConversation(headMessages, thread)` — pure renderer that walks the head in order and produces the single `# Past Conversation` string, grouped by turn:
  - `## Turn <N>` header whenever the turn metadata changes; speaker-labelled lines separated by `--`:
    - voiced seat (assistant) → `You: <text>`.
    - counterpart (user) → `<Role> of <Civ>: <text>` from the thread's stored audience role + identity (reuse `roleOf` / `identityOf`, as `formatUserDescription` does).
  - `deal` rows render **by type**, because only proposals carry terms:
    - **proposal / counter** rows carry the full `Deal` → render through the **shared `formatDealBlock`** (below), identical to the live on-the-table view, no bespoke summarizer.
    - **accept / reject / enacted** rows carry only a `Payload.ProposalMessageID` reference, *not* terms. Resolve it against the proposals already walked in the head: keep a `Map<proposalID, proposalRow>` (every proposal/counter row registers its `ID`; reuse `deriveActiveProposal`'s `answeredProposalID` to read the reference) and emit a one-line status event naming the resolved proposal — e.g. `<Speaker> accepted the deal (msg #N)`. Terms are **not** reprinted (they already appear once, at the proposal/counter row being answered). A reference with no matching proposal in the head (shouldn't happen) degrades to `<Speaker> accepted a deal`.
  - assistant verbose trails (reasoning + tool-call parts) → collapsed to their **spoken result** via the **same `collectSpokenReply` join 05.1 uses for archival**, so the head line equals the re-hydrated `text` row by construction. In the live tail an assistant turn is a `send-message` tool call (possibly preceded by narrated free text, which 05.1 also archives); in the head it renders as a single `You: <text>` line — the in-order concatenation of its `text` parts and `send-message` `Message` input. The durable store holds that same concatenation as one plain `text` row, so a reload yields the already-compacted form and the line is **byte-stable across the reload boundary**. Because every head line is byte-stable this way, when the tail folds into the head on refresh the head is a clean *extension* of the previously-cached prefix (one re-cache), never a rewrite that busts it. (The negotiator handoff already leaves a `deal` row, rendered separately.)
- `splitForCompaction(messages, boundary)` → `{ head, tail }`: `head = messages[0..boundary]`, `tail = the rest`. (No token counting; the boundary is the fixed hydrated length.)

## Reuse, don't reinvent the deal format

Extract the single-deal block out of `formatDealContext` (`diplomat-utils.ts`) into a shared, exported `formatDealBlock(deal, messageType, status, viewerID, civName, inspection?)` (terms via `formatDealTermsByDirection` + a status/header line + the one-line `message`). `formatDealContext` (the live current-deal view) calls it for the active proposal; the compaction renderer calls it for each historical **proposal/counter** `deal` row (omitting the live-only `inspection` / promise-agreeability and inward rationale). Historical accept/reject/enacted rows are *not* passed to `formatDealBlock` (they carry no terms) — they render as the one-line status events described above, resolved to the proposal they answer. So a proposal reads the same in `# Past Conversation` as on the table, and its later acceptance/rejection reads as a short event referencing it.

## Integration

- **`vox-agents/src/utils/diplomacy/transcript.ts`** — `syncThreadMessages`: after rebuilding `thread.messages`, record `boundary = thread.messages.length` for `thread.id` (the single hydration point that seeds "everything before the active thread").
- **`vox-agents/src/envoy/live-envoy.ts`** `getInitialMessages` (normal mode only): replace the `convertToModelMessages(filterSpecialMessages(input.messages))` block with `splitForCompaction` → push the `# Past Conversation` user message (with the `cacheControl` breakpoint) → push `convertToModelMessages(tail)` (existing per-message conversion still handles turn prefixes, reasoning drop, and the `send-message` tool-result skip from 05.1). Special/greeting mode keeps skipping history.
- The Diplomat's `getInitialMessages` deal-context append and the hint/addon are unchanged — they trail the tail and stay uncached (correct: turn- and deal-state-specific).

## Verification
1. `cd vox-agents && npm run build` and `npx vitest run tests/mock/envoy tests/mock/utils`.
2. In a long multi-turn diplomacy chat with several proposals/counters, inspect the assembled prompt (telemetry span `step.messages`): the settled history is a single `# Past Conversation` block with readable deal summaries and a cache breakpoint, and only the recent exchanges trail it as live messages.
3. Reload the conversation and re-inspect. Reopened at the **same** durable state, the `# Past Conversation` block is byte-identical (the provider cache still hits). If new turns were sent during the session, the block has **extended** by exactly those turns — earlier lines unchanged, one re-cache of the longer prefix — and a turn that contained a `send-message` reply renders the **same** `You: <text>` line it streamed, so the extension is a clean prefix, never a rewrite.
