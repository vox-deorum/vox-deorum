# Stage 6 — civ5-dll + mcp-server: enact the agent deal (the only gameplay change)

> Part of the interactive-diplomacy plan. Shared design and watch-items live in [README.md](README.md); requirements in [specs.md](specs.md).

## Objective

Flip on **real enactment**: a complete deal object — ordinary **trade items** and the **nine promises** — is enacted for real in the game, bypassing the AI's political refusal while honoring structural legality (specs §3, §4). This is the **only gameplay code change** in the feature and the last piece of human↔LLM Web v1. `CvDealAI` valuation is left **completely untouched** and the normal in-game deal pathway **behaves exactly as before**: every DLL edit is a *defaulted* signature extension (stock callers reproduce stock behavior exactly), plus two thin additive Lua bindings.

## Approach: thread the override through the existing pipeline; add only thin bindings

Everything enactment needs already exists — it is just not reachable in override form. The DLL already has a complete, standalone-deal-safe finalize chain (`FinalizeMPDeal` → validate via `AreAllTradeItemsValid` → `FinalizeDealValidAndAccepted` → `ActivateDeal`, with failure logging, notifications, renew-ID reset, and UI dirty-bit — routinely called with deals never on the proposed-MP list); the promise setters already stamp their own turn and carry their reciprocal side-effects; and `inspect-deal.lua` already builds the scratch `CvDeal` from the structured deal schema and computes per-term legality under the stage-3 `bTreatAsHumanToHuman` override. So instead of a monolithic new entrypoint that re-implements build + validate + activate + promises in C++, this stage **threads the stage-3 override through the existing construction/validation/finalize functions as defaulted parameters** (the fork's established convention) and adds exactly two thin Lua bindings: `Deal:Enact()` (a five-line wrapper over `FinalizeMPDeal`) and `Player:SetPromise(...)` — the validation reads reuse already-exposed endpoints (`GetNumTurnsXxxPromise`, `GetCoopWarAcceptedState`).

On the mcp-server side, the same philosophy: `inspect-deal.lua` gains an **opt-in enact mode** (an optional argument; absent reproduces today's read-only behavior, so the stage-3/4 knowledge-tool callers are untouched). Because a bridge Lua script executes synchronously to completion on the game thread, the whole validate→enact sequence runs in **one atomic invocation** — validation cannot go stale between check and act, which a TS-side check followed by a second bridge call could not guarantee. Keep the DLL/tool split explicit: the DLL surface stays a pure, stateless game operation (no transcript, proposal IDs, or UI state); the **mcp-server `enact-agent-deal` tool** (shipped as a transcript-only **stub** in stage 5) remains transcript-aware — idempotency on a prior `deal-enacted`, then the real enactment, then the atomic `deal-accept` + `deal-enacted` writes. This stage drops the real game operation into that stub rather than building a new tool.

## Work items

1. **`civ5-dll/.../CvDealClasses.cpp/.h` — validation threading.** Add the defaulted `bTreatAsHumanToHuman = false` to `AreAllTradeItemsValid` (threaded into the 9th arg of its internal `IsPossibleToTradeItem` call) and to all 17 `CvDeal::Add*` trade-item constructors, threaded into each one's internal `IsPossibleToTradeItem` guard — today those guards evaluate the *computed* classification and silently refuse override-only items, so agent items couldn't even be added. Mechanical care: the new param goes **after** `bDoNotRemove` (defaults `true`), so positional C++ callers are untouched, and the guard calls must spell out the intermediate defaulted args (`iData2/iData3/bFlag1/bFinalizing`) to reach position 9. Default-`false` everywhere preserves the stock path; passing `true` evaluates the structural guards in their most permissive form, so AI-only restrictions (one city per player, no peacetime selling of self-founded cities, the `DEALAI_DISABLE_CITY_TRADES` toggle, and other `!bHumanToHuman` gates) don't gate agent deals, while the always-on guards (ownership, quantity, capital, duplicate-luxury, banned luxuries, embassy-for-city, sapped/damaged cities) still apply (specs §4).
2. **`CvDealClasses.cpp/.h` — finalize threading.** The same defaulted param on `FinalizeMPDeal` → `FinalizeDealValidAndAccepted` → `ActivateDeal`, consumed in exactly two places: the `AreAllTradeItemsValid` call and the peace-surrender assignment (`bShouldSetHumanSurrender`) — an agent peace deal is built from scratch, so without the override it would activate with no surrendering player. It must **not** touch `ActivateDeal`'s observer/debug notification gates (`… && !bHumanToHuman`): forcing the override there would suppress observer notifications for agent-brokered third-party wars/peace. All existing callers pass the old arities and are unchanged.
3. **`Lua/CvLuaDeal.cpp` — wrappers + one new binding.** A trailing `luaL_optbool(L, n, false)` on the 17 `lAdd*` wrappers (the stage-3 optional-9th-arg pattern; the game's own TradeLogic.lua callers pass fewer args and keep stock behavior). New `lEnact` (`Deal:Enact()`), registered in `PushMethods`: pre-clear the renew-deal ID (`SetRenewDealID(from, to, -1)`) so an unrelated pending renewal for the pair is never clobbered by `ActivateDeal`'s type+direction renewal diff, then `FinalizeMPDeal(*pkDeal, /*bAccepted*/ true, /*bTreatAsHumanToHuman*/ true)` — acceptance pre-decided, **never** invoking `CvDealAI` — returning the bool. No runtime `MOD_ACTIVE_DIPLOMACY` gate on the bindings (stage-3/4 convention: plain additive registration; `FinalizeMPDeal` itself already lives in the `MOD_ACTIVE_DIPLOMACY` region).
4. **`Lua/CvLuaPlayer.cpp` — promise binding.** `lSetPromise(fromPlayerID, kind, targetID?, apply)`, called on the promise **recipient** (promise state lives on the recipient's `CvDiplomacyAI`, keyed by giver), dispatching over **all nine kinds** — forward-compatible beyond the five the shipped ledger currently authorizes — to the existing setters, which stamp their own turn and carry their reciprocal side-effects; plus the stock companion calls the accept flow performs outside the setters: Spy → `EvaluateSpiesAssignedToTargetPlayer`; No-Convert / No-Digging → `SetPlayerAskedNotToConvert` / `SetPlayerAskedNotToDig`; **Coop War** → the full stock accept block: paired `SetCoopWarState(…, COOP_WAR_STATE_PREPARING)` on **both principals**, `SetCivApproach(target, CIV_APPROACH_WAR)`, and `ChangeRecentAssistValue(+300)`. `apply = false` clears (`NO_PROMISE_STATE` / `NO_COOP_WAR_STATE`) for rollback — the setters reverse cleanly (turn stamps and asked-not-to flags reset). Returns **previous + new state** — one return covering both silent setter refusal (e.g. the military setter's vassal / mutual-human guards) and already-`MADE` detection. **No read binding is added**: existing endpoints already cover pre-write validation where it matters (`GetNumTurnsMilitary/Expansion/BorderPromise >= 0` reads exactly "state == `MADE`" — they return -1 when not made; `GetCoopWarAcceptedState` reads the coop-war state), and the kinds with no exposed made-read (No-Digging today; the four dormant kinds later) are covered by the previous-state return at apply time. Speak kind strings and apply/clear booleans over the boundary — `PromiseStates` is not a Lua-exposed enum. Because every promise writes state the game already persists, **no new save fields** are introduced, and honoring/expiry is governed by the game's existing `CvDiplomacyAI` timers and break-detection (e.g. a later DoW breaks the military promise).
5. **`mcp-server/lua/inspect-deal.lua` + `src/utils/lua/inspect-deal.ts` — opt-in enact mode.** An optional enact argument carrying the promise commitments (absent → today's read-only behavior — the same defaulted-parameter philosophy). Enact mode, inside the one atomic invocation: **(a) validate everything before any write** — build the scratch deal via the override-aware `Add*` calls (detect a silent constructor refusal via `GetNumItems` deltas; keep `hasUnresolvedCityCoordinates`), run the existing `legalityOf` loop over the items, and validate every promise (distinct living majors; **not already made** via the existing reads — `GetNumTurnsXxxPromise >= 0` for military/expansion/border, `GetCoopWarAcceptedState` for coop war; valid Coop-War target via the existing `coopWarEligible`; dedupe the symmetrized COOP_WAR twin rows) — refusing with per-term reasons; **(b) apply promises** via `SetPromise` with read-back, the side-effect-heavy Coop War last — a returned previous state of `MADE` (the kinds without an exposed read) refuses too — rolling back on any failure **only** the promises whose returned previous state was clear, so a pre-existing promise is never wiped; **(c) `deal:Enact()`** — the irreversible step last; its items were pre-validated in the same invocation so failure is unreachable, but on `false` roll back the promises and refuse. `ClearItems()` on every exit path; skip the range/promiseTargets enumeration in enact mode; update the script's read-only header contract.
6. **`mcp-server/src/tools/actions/enact-agent-deal.ts` — fill the stage-5 stub.** Keep its serialized validation/idempotency boundary and atomic `deal-accept` + `deal-enacted` writes; at the marked insertion point, replace the read-only `inspectDeal` re-check with the enact-mode call carrying the stored deal's items **and** promises, and flip `Enacted` to `true` on success (it stays `false` on the `AlreadyEnacted` idempotent path — "not enacted by this call"). **Invert the stub's bridge-failure policy**: a bridge error or un-enacted result now throws and writes nothing — the stub's lenient fall-through was correct for a redundant re-check but would here record `deal-enacted` with no in-game effect and permanently block retry via idempotency. Watch-item: a DB failure *after* a successful enact leaves an enacted deal without its record (a retry would double-enact) — accepted; the write is the next statement and the `DealMade` IPC event is the reconciliation signal. The tool's external contract (input: full `Deal` + proposal message ID; output records) is unchanged from the stub; the `deal-enacted` message remains the idempotency record — no status column, no separate action/replay log.
7. **Rebuild the DLL — no version bump.** No `TradeableItems` addition, no enum/save/serialization change, no new acceptability/valuation logic: defaulted signature extensions plus two additive bindings only.

Caller orchestration after acceptance is unchanged from the stub design: Web UI / vox-agents reduce the append-ordered transcript to decide *which* proposal is the current accepted one, then call `enact-agent-deal` with that proposal's deal object and message ID. The UI may disable the submit control for UX, but duplicate-enact correctness is enforced by the tool, not the caller.

## Reuse

The full finalize chain `FinalizeMPDeal` / `FinalizeDealValidAndAccepted` / `ActivateDeal` with its failure logging, notifications, renew-ID reset, and UI dirty-bit (`CvDealClasses.cpp`); stage-3's defaulted `bTreatAsHumanToHuman` pattern, extended to `AreAllTradeItemsValid` and the `Add*` constructor family; the existing `bHumanToHuman` classification and `IsPossibleToTradeItem` guards; the eight `SetXxxPromiseState` setters (internal turn-stamping + side-effects), `SetCoopWarState` / `COOP_WAR_STATE_PREPARING`, and the stock accept flow's companion calls (`CvDiplomacyAI.*`, `CvDiplomacyAIEnums.h`); the already-exposed promise reads (`GetNumTurnsMilitary/Expansion/BorderPromise`, `GetCoopWarAcceptedState`) and the stage-4 `IsValidCoopWarTarget` binding (the screen's eligibility preview and enactment's coop-war check stay aligned); `inspect-deal.lua`'s `resolveItem` / `legalityOf` / `coopWarEligible` and the `inspectDeal()` TS helper (`LuaFunction` pattern); the stage-5 `enact-agent-deal` tool's idempotency/transaction boundary and the stage-1 transcript store.

## Verify

Via `lua-executor` and end to end through stages 2–5: enact a complete deal object the **stock AI would refuse on political grounds** (e.g. two cities from one side — an AI-only `!bHumanToHuman` gate) — items change hands for real and the `DealMade` IPC event fires. A structurally-illegal item (capital, unowned resource) is rejected with a reason and **nothing** is written — no promise state, no transcript records. A deal with valid items and one bad promise (e.g. Coop War against an ineligible target) rejects **before any trade item changes hands**, with earlier-applied promises rolled back (state and turn stamps restored). A promise already `MADE` for the pair (e.g. No-Digging, detected by `SetPromise`'s previous-state return) refuses the same way — without wiping the pre-existing promise. An enacted promise thereafter **behaves like an in-game promise** — a later declaration of war breaks the military promise; a Coop War deal shows `PREPARING` in both directions with the approach/assist side-effects applied. A lopsided agent **peace** deal gets a surrendering player assigned; an agent **third-party war** deal still produces observer notifications (the override must not leak into the notification gates); a pending deal **renewal** for the pair is not clobbered. End to end, `enact-agent-deal` records one `deal-enacted` message on success and **refuses a second enactment of the same proposal ID**; with the bridge down it errors and writes nothing. The **normal in-game deal pathway and AI valuation behave exactly as before** — all extensions are defaulted (human trade screen and AI–AI turn deals unchanged, pre-change saves load) — and there is no `TradeableItems`/save-format change.

## Done when

Both sides agreeing in a Web conversation results in the accepted deal object being **enacted for real** — trade items and any of the nine promises — through the override-threaded existing pipeline and its thin additive bindings, with structurally-illegal items still rejected, a `deal-enacted` archive message recorded by `enact-agent-deal` on success (and a second enactment of the same proposal refused), and the stock pathway untouched. **This completes human↔LLM Web v1** (specs § Success criteria).
