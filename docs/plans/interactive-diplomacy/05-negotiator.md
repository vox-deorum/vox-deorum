# Stage 5 — vox-agents: negotiator agent + diplomat deal context + the `call-negotiator` handoff

> ✅ **DONE.** Part of the interactive-diplomacy plan. Shared design and watch-items live in [README.md](README.md); requirements in [specs.md](specs.md).
>
> 🔁 **Refactored (post-stage):** the bespoke loop driver was removed in favour of the codebase's normal agent-handoff pattern. The diplomat now hands off through the single auto-registered **`call-negotiator`** tool (like `call-diplomatic-analyst`), and the negotiator is fully self-contained — it reads the transcript to choose its task, inspects upfront itself, and returns its own summary. See **The `call-negotiator` handoff** below.

## What was built (summary)

- **`vox-agents/src/envoy/negotiator.ts`** — the self-contained `Negotiator` agent and its three terminal tools (`accept-deal`, `propose-deal`, `reject-deal`). It extends **`VoxAgent` directly** (like `DiplomaticAnalyst`) rather than `LiveEnvoy`: `LiveEnvoy`/`Envoy` are typed `TInput = EnvoyThread`, but the negotiator takes a structured `NegotiatorInput`, so extending the `EnvoyThread`-typed base would have forced overriding nearly everything and fighting the input typing. It still reuses the same building blocks — `buildGameContextMessages` + `createBriefingTool` + `get-diplomatic-events` — so the spec's "grounded by three contexts" intent (work item 1) is fully met. It is the sole decider of terms; the terminal tools persist the move through the durable store and record it on the shared `NegotiatorInput.outcome`, which `getOutput` summarizes for the diplomat. It exposes a small caller-facing **`handoffSchema`** (`{ Briefing, Intent? }`); **`resolveHandoffInput`** folds in the ambient conversation thread; **`resolveHandoffTarget`** dispatches to the voiced seat's configured negotiator (default `negotiator`). In `getInitialMessages` it reduces the transcript to decide its task — respond to a still-open counterpart proposal, else open one — and runs `inspect-deal` upfront itself.
- **`vox-agents/src/envoy/diplomat-deal-tools.ts`** + edits to **`diplomat.ts`** — the single `call-negotiator` handoff in the diplomat's active tools (gated to diplomacy threads), the prompt section describing it, and deal-context injection in `getInitialMessages` so the diplomat "sees the deal at every step" (active proposal terms + outward message + per-item value snapshots + status). Its own negotiator's rationale is included; an opposing negotiator's private rationale is not. (This module now holds only the deal-context renderer; the relay tools are gone.)
- **`vox-agents/src/utils/tools/agent-tools.ts`** + **`vox-agents/src/infra/vox-agent.ts`** — the reusable handoff seam. `createAgentTool` exposes `handoffSchema ?? inputSchema`, maps the caller's args through `resolveHandoffInput`, and executes the agent named by `resolveHandoffTarget` (defaults to the agent itself). This lets any `call-<agent>` handoff enrich the LLM's arguments with ambient context and dispatch to a context-resolved variant.
- **`vox-agents/src/utils/diplomacy/resolve-negotiator.ts`** — `resolveNegotiator(thread)`: resolves the voiced seat's negotiator (duck-typed off the active session's `getPlayerAssignments`, default `negotiator`, falling back when the configured agent is not registered). Used by `Negotiator.resolveHandoffTarget`. Kept in its own module so importing the session/agent registries does not cycle with the agent registry.
- **`vox-agents/src/utils/diplomacy/deal-reduce.ts`** — the server-side deal-state reducer (twin of the stage-4 UI reducer) over `TranscriptMessage[]`, plus `enactAgentDeal` / `readActiveProposal` helpers in `deal.ts`.
- **`mcp-server/src/tools/actions/enact-agent-deal.ts`** — a **stub** enactment route (per the stage-5 design decision): the sole writer of `deal-accept` + `deal-enacted`, idempotent on the proposal's `deal-enacted` record, but it does **not** apply in-game effects — `Enacted: false`. It accepts only the current open proposal, only from its non-authoring recipient, validates the stored terms, and performs its check plus both transcript writes in one serialized transaction. The Web accept route and the negotiator's `accept-deal` both route through it. **Stage 6 inserts the DLL validation + enactment call inside this same tool** (between the idempotency check and the writes), keeping the external contract identical.
  - **Contract note:** because the stub records both `deal-accept` and `deal-enacted`, the reducer reports status `enacted` after acceptance (agreement reached + orchestration recorded); only the *in-game* transfer is still pending for stage 6.
- **Tests:** `deal-reduce`, `enact-io`, `negotiator` (terminal tools + handoff mapping/target + task determination + summary), `diplomat-deal-tools`, `agent-tools` (handoff seam), and Web accept-route coverage (vox-agents); `enact-agent-deal` concurrency, rollback, lifecycle, participant, and payload coverage (mcp-server). ## Objective

Deals move through the **diplomat→negotiator handoff** until both sides agree (specs §3, §7). An LLM civ's diplomacy uses **two cooperating agents**: a **diplomat** (the human's only conversational counterpart — free-text, owns the thread) and a **negotiator** (a deal specialist that never reads human free-text). The **negotiator is the sole decider of deal terms**; the **diplomat only relays context** and never authors terms.

A deal needs shaping when the human proposes one or the diplomat decides it is time to negotiate. The diplomat hands the conversational context (a briefing, plus an optional intent when opening a deal) to the negotiator through the single **`call-negotiator`** tool. The negotiator reads the transcript itself — responding to a still-open counterpart proposal, otherwise opening one — inspects the tradable range + per-term estimates upfront (`inspect-deal`), then chooses **exactly one** of three terminal tools — accept-as-is / propose-or-counter / reject-as-is — and returns its summary to the diplomat, which voices it on the Web.

Real enactment still lands in stage 6, so the milestone here is **a deal both sides agree on**, reached through the handoff and surfaced on the Web — not items changing hands.

## Work items

1. **`vox-agents/src/envoy/negotiator.ts`** — a deal specialist extending `VoxAgent`, invoked by the diplomat through the **`call-negotiator`** agent-tool, the **sole decider of deal terms**. It never reads the free-text thread and is grounded by **three contexts**, all assembled by the agent itself:
   - **(1) game context + strategy/persona** — `get-briefing` / `get-diplomatic-events` plus the diplomat's briefing (the handoff `Briefing`);
   - **(2) what is tradable and each item's value** — the `inspect-deal` results (tradable range + per-term legality + both-direction value/agreeability), **run upfront in `getInitialMessages`** (the model does not call `inspect-deal` itself);
   - **(3) what is on the table** — the active proposal, **derived from the transcript** in `getInitialMessages` (a still-open counterpart proposal → respond; otherwise open one using the optional `Intent`).
   It carries a **promise-aware deal artifact** (trade items + promise commitments — specs §3) and chooses **exactly one** of three terminal tools, each returning an inward **`rationale`** (the negotiator's reasoning for the diplomat, never voiced verbatim):
   - **accept the deal as-is** — `rationale` only; enactment routes through `enact-agent-deal` (stage 6);
   - **propose / counter** — also writes a one-sentence outward **`message`** (schema-constrained to one sentence), storing both on the draft deal (`Payload.Deal.rationale` / `.message`); the draft becomes a `deal-proposal` or `deal-counter`;
   - **reject the deal as-is** — `rationale` only.
   Authority is **baked into the chosen negotiator agent** (specs §7), no separate ratification knob.
2. **`vox-agents/src/envoy/diplomat.ts`** — the existing `Diplomat` lists the single **`call-negotiator`** handoff in its active tools (gated to diplomacy threads, alongside the `close-conversation` tool from stage 2). The diplomat **never authors deal terms** — it hands the conversational context in (always a `Briefing`; an `Intent` when it wants to open a deal — no terms) and voices the negotiator's move out. It **sees the deal at every step** — each deal move enters the diplomat's model context with its outward `message`, terms, and per-term estimates; its own negotiator's inward `rationale` is included, while the opposing side's private rationale is not — so it voices each move faithfully and keeps gathering intelligence.
3. **The `call-negotiator` handoff** — the reusable agent-handoff seam (`vox-agents/src/utils/tools/agent-tools.ts` + `infra/vox-agent.ts`) lets the diplomat hand off through the auto-registered `call-negotiator` tool (the same mechanism as `call-diplomatic-analyst`): `handoffSchema` is the small caller-facing schema, `resolveHandoffInput` enriches it with the ambient thread (still the diplomat's `currentInput` at call time), and `resolveHandoffTarget` dispatches to the **voiced seat's** configured `negotiator` (the `negotiator` field from the seat config added in stage 2; default `negotiator`). Register the agents in `vox-agents/src/infra/agent-registry.ts` (the `initializeDefaults` idiom). **No blocking, human-paced assumption:** the negotiator relays an already-present deal (or an intent) and returns synchronously once it decides; the human→LLM path rides the existing pause (specs §8), and stage 8's LLM↔LLM peers reuse this same handoff rather than forking it.
4. **Deal-state reducer over the transcript** — implement the server-side equivalent of the stage-4 UI reducer over append-ID order: proposal/counter messages create the active proposal, accept/reject messages reference the proposal they answer, `deal-enacted` records successful orchestration once stage 6 exists, and agreement exists only when the current proposal has the required acceptance from the recipient and no later counter/reject supersedes it. This reducer is used by Web/agent orchestration before calling stage-6 enactment; there is still no status column.
5. **Stored deal moves in the transcript** — proposal and counter messages carry the proposed terms in `Payload.Deal`, including the optional **`Payload.Deal.rationale`** and **`Payload.Deal.message`** (both optional — pinned in stage 3) and optional proposal-time `Payload.Value1` / `Payload.Value2` snapshots. Accept/reject/enacted messages carry no draft deal — just `Payload.ProposalMessageID`. Current legality and enactment checks still come from fresh `inspect-deal` / `enact-agent-deal` calls; `enact-agent-deal` is idempotent (it refuses a second enactment of a proposal ID that already has a `deal-enacted`).

## Reuse

`LiveEnvoy` / `Envoy` and the existing `Diplomat` (`src/envoy/*.ts`); the agent-as-tool handoff pattern already used for `call-diplomatic-analyst` (extended here with `handoffSchema` / `resolveHandoffInput` / `resolveHandoffTarget`); `createBriefingTool` and `get-diplomatic-events`; stage-3 `inspect-deal`; stage-2's deal-message flow and the Web deal screen (stage 4) as the human's proposal/counter surface.

## Verify

In a live session: (a) **diplomat-decided path** — the diplomat calls `call-negotiator` with an intent + briefing (no deal on the table); the negotiator runs `inspect-deal`, values from that context, builds terms with a `rationale` + one-sentence `message` on the draft deal, and the diplomat voices it on the Web; the per-term estimates are visible to both agents in their reasoning/logs. (b) **human-forwarded path** — the human assembles a deal on the stage-4 screen (with an attached `message`); the diplomat calls `call-negotiator` with a briefing; the negotiator sees the on-the-table proposal in the transcript and counters with a draft deal carrying its `rationale` **and** one-sentence `message`; the diplomat relays the counter and its estimates; the human accepts and the transcript reducer reports an **agreed** deal. (Enactment is asserted in stage 6.)

## Done when

Both the diplomat-decided and human-forwarded paths run end to end through the `call-negotiator` handoff — the diplomat relays context in; the negotiator reads the transcript to pick its task, inspects, values, and chooses exactly one of accept / propose-or-counter / reject, attaching a `rationale` (and, on propose/counter, a one-sentence `message`) to the draft deal; the diplomat voices every move with its per-term estimates; and the append-only transcript reduces to an agreed current proposal — with real enactment the only thing still pending.
