# Stage 1 — mcp-server: durable conversation transcript store

> Part of the interactive-diplomacy plan. Shared design and watch-items live in [README.md](README.md); requirements in [specs.md](specs.md).

## Objective

Conversation transcripts become **durable** in the mcp-server, surviving a restart, keyed by the game and a **player pair ordered by `playerID`**. There is exactly **one conversation per pair of major-civ players**, so the store needs no thread identity, no thread table, and no status column (specs §6) — the conversation *is* the ordered list of messages between the two players. This stage stands alone at the storage layer: it is verified entirely through tool calls, with no agent or Web changes yet.

The store holds **only the messages** — the two players, each player's role, the speaker, content, message type, turn, timestamp, and optional message metadata in `Payload`. It does **not** store LLM internals (reasoning, scratch state, tool traces), which stay transient in vox-agents. Deal proposal and counter messages store the proposed terms directly in `Payload.Deal`, plus optional proposal-time `Payload.Value1` / `Payload.Value2` snapshots for the two ordered players. Human-side values are left undefined. Accept/reject/enacted messages carry `Payload.ProposalMessageID`, pointing at the proposal they answer or record as enacted. Legality is not stored; the proposal exists because it was put in the conversation, enactment re-validates live, and `deal-enacted` records successful orchestration rather than DLL state.

## Work items

1. **`mcp-server/src/knowledge/schema/timed.ts`** — a `DiplomaticMessage` interface extending `TimedKnowledge`, with `Player1ID` / `Player2ID` (`min` / `max` `playerID`), `Player1Role` / `Player2Role` (`human` / `llm` / `observer`, remapped to match the ordered IDs), `SpeakerID`, `MessageType` (`text`, `close`, `deal-proposal`, `deal-counter`, `deal-accept`, `deal-reject`, `deal-enacted`), and `Content`. Use the inherited `Payload` JSON for optional message metadata. Proposal and counter messages store terms at `Payload.Deal`, and optional proposal-time estimates at `Payload.Value1` / `Payload.Value2`. Accept/reject/enacted messages store `Payload.ProposalMessageID`. Do not put LLM internals in `Payload`. Export the type from `schema/index.ts` and add `DiplomaticMessages` to the `KnowledgeDatabase` interface in `schema/base.ts`.
2. **`mcp-server/src/knowledge/schema/setup.ts`** — create the table in `setupKnowledgeDatabase()` via `createTimedKnowledgeTable(db, 'DiplomaticMessages')` with the columns above, the standard timed indexes, and an extra **player-pair index** on `(Player1ID, Player2ID, ID)` for transcript reads. No status columns — this is `TimedKnowledge`, not `MutableKnowledge`. `Turn` remains metadata; append `ID` is canonical transcript order.
3. **`mcp-server/src/tools/actions/append-message.ts`** (new) — a non-read-only `ToolBase` write tool taking endpoint-oriented input such as `{ PlayerAID, PlayerBID, PlayerARole, PlayerBRole, SpeakerID, MessageType, Content, Payload?, Turn? }`, then ordering the two player IDs server-side and remapping roles to `Player1Role` / `Player2Role` before storage. Default `Turn` to the current server turn (`knowledgeManager.getTurn()`, the idiom `PresentDecisionTool` uses) when omitted; Web UI callers should omit `Turn`, leaving the parameter for future/internal use. Validate that the two endpoints are distinct, that `SpeakerID` is one of them, and that each non-observer endpoint is a living major when game state is available. The **observer sentinel (`-1`)** is a special case: it is accepted (not rejected), sorts to `Player1ID`, takes the `observer` role, and is exempt from the living-major check; the *other* endpoint must still be a living major. Write one row via `store.storeTimedKnowledgeBatch('DiplomaticMessages', [...])`, set `Payload` to `{}` when omitted, and pass visibility flags for the real participant(s) — both for a civ↔civ conversation, and only the real civ when the counterpart is the observer (which has no player slot). `append-message` is archival only: it does not stream, notify, run agents, enact deals, or decide whether a deal is current/accepted. Register the factory in `tools/index.ts`.
4. **Message-specific validation in `append-message`** — for `deal-proposal` and `deal-counter`, validate that `Payload.Deal` exists; `Payload.Value1` and `Payload.Value2` are optional snapshots and should be undefined for human participants. For `deal-accept` and `deal-reject`, validate that `Payload.ProposalMessageID` references an earlier proposal/counter message between the same players and that `SpeakerID` is the opposite endpoint from the referenced proposal/counter speaker. The **close-conversation special message** is just an append with the close `MessageType` (no separate mechanism). `deal-enacted` is **not** emitted through this public tool: it is written only by `enact-agent-deal` (stage 6) via the same store path, authored by the enacting party, so the idempotency check (no prior `deal-enacted` for the proposal ID) and the record are one operation — it carries `Payload.ProposalMessageID` and stores no legality, reasons, or DLL state. The schema `MessageType` column still includes `deal-enacted` because those rows live in the same table.
5. **`mcp-server/src/tools/knowledge/read-transcript.ts`** (new, with a getter under `knowledge/getters/`) — a read tool taking `{ PlayerAID, PlayerBID }`, deriving the same ordered player pair, and returning messages ordered by append `ID` so both directions read as a single thread. The query filters by player pair rather than by one speaker's visibility; visibility remains useful for generic knowledge readers and future UI filtering. Register it in `tools/index.ts`.

## Reuse

`createTimedKnowledgeTable` / `createTimedKnowledgeIndexes` (`schema/setup.ts`, `schema/table-utils.ts`); `storeTimedKnowledgeBatch` (`knowledge/store.ts`); the `ToolBase` read/write shape and getter idiom modeled on `get-opinions` / `get-players`; the `toolFactories` registry in `tools/index.ts`.

## Verify

Via MCP client tool calls against a running game (no agent involvement): `append-message` several messages in both directions between two civ `playerID`s (and one close-conversation special message); append a deal proposal and an accept message from the opposite endpoint that references it. Confirm an accept/reject from the same speaker as the referenced proposal is rejected (`deal-enacted` is exercised in stage 6, not here). Append one message with the observer sentinel (`-1`) as a counterpart and confirm it is stored with the `observer` role, sorted to `Player1ID`, with visibility set only for the real civ. `read-transcript` returns messages in append-ID order as one thread whether requested as `{ PlayerAID, PlayerBID }` or `{ PlayerBID, PlayerAID }`. Stop and restart mcp-server, re-read — the transcript persists. Confirm no thread/status table was created, no live action was triggered by append alone, and both participant visibility flags are set on every civ↔civ row.

## Done when

A conversation between two civs can be appended to and read back as one ordered, durable thread keyed only by game + the player pair ordered by `playerID`, surviving a restart — exercised purely through the new tools.
