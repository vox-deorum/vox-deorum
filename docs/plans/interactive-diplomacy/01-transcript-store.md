# Stage 1 — mcp-server: durable conversation transcript store

> Part of the interactive-diplomacy plan. Shared design and watch-items live in [README.md](README.md); requirements in [specs.md](specs.md).

## Objective

Conversation transcripts become **durable** in the mcp-server, surviving a restart, keyed by the game and a **player pair ordered by `playerID`**. There is exactly **one conversation per pair of major-civ players**, so the store needs no thread identity, no thread table, and no status column (specs §6) — the conversation *is* the ordered list of messages between the two players. This stage stands alone at the storage layer: it is verified entirely through tool calls, with no agent or Web changes yet.

The store holds **only the messages** — the two players, each player's role, the speaker, content, message type, turn, timestamp, and optional message metadata in `Payload`. It does **not** store LLM internals (reasoning, scratch state, tool traces), which stay transient in vox-agents. Deal proposal and counter messages store the proposed terms directly in `Payload.Deal`, plus optional proposal-time `Payload.Value1` / `Payload.Value2` snapshots for the two ordered players. Human-side values are left undefined. Accept/reject/enacted messages carry `Payload.ProposalMessageID`, pointing at the proposal they answer or record as enacted. Legality is not stored; the proposal exists because it was put in the conversation, enactment re-validates live, and `deal-enacted` records successful orchestration rather than DLL state.

## Work items

1. **`mcp-server/src/knowledge/schema/timed.ts`** — a `DiplomaticMessage` interface extending `TimedKnowledge`, with `Player1ID` / `Player2ID` (`min` / `max` `playerID`), `Player1Role` / `Player2Role` (**free-form role descriptors** following the EnvoyThread tradition — an agent name for an LLM-voiced side such as `diplomat` / `spokesperson` / `negotiator`, the `UserIdentity.role` text for a human side such as `the leader`, or `observer` for the observer endpoint; remapped to match the ordered IDs; they do **not** encode human-vs-LLM), `SpeakerID`, `MessageType` (`text`, `close`, `deal-proposal`, `deal-counter`, `deal-accept`, `deal-reject`, `deal-enacted`), and `Content`. Use the inherited `Payload` JSON for optional message metadata. Proposal and counter messages store terms at `Payload.Deal`, and optional proposal-time estimates at `Payload.Value1` / `Payload.Value2`. Accept/reject/enacted messages store `Payload.ProposalMessageID`. Do not put LLM internals in `Payload`. Export the type from `schema/index.ts` and add `DiplomaticMessages` to the `KnowledgeDatabase` interface in `schema/base.ts`.
2. **`mcp-server/src/knowledge/schema/setup.ts`** — create the table in `setupKnowledgeDatabase()` via `createTimedKnowledgeTable(db, 'DiplomaticMessages')` with the columns above, the standard timed indexes, and an extra **player-pair index** on `(Player1ID, Player2ID, ID)` for transcript reads. No status columns — this is `TimedKnowledge`, not `MutableKnowledge`. `Turn` remains metadata; append `ID` is canonical transcript order.
3. **`mcp-server/src/tools/actions/append-message.ts`** (new) — a non-read-only `ToolBase` write tool taking endpoint-oriented input such as `{ PlayerAID, PlayerBID, PlayerARole?, PlayerBRole?, SpeakerID, MessageType, Content, Payload?, Turn? }`, then ordering the two player IDs server-side and remapping the free-form roles to `Player1Role` / `Player2Role` before storage. Default `Turn` to the current server turn (`knowledgeManager.getTurn()`, the idiom `PresentDecisionTool` uses) when omitted; Web UI callers should omit `Turn`, leaving the parameter for future/internal use. Validate that the two endpoints are distinct, that `SpeakerID` is one of them, and that each non-observer endpoint is a **major civilization** — checked against the **cached `PlayerInformations`** (`readPublicKnowledgeBatch`, which only falls back to a live fetch when the cache is empty), not a live per-append game query — when player information is available. The **observer sentinel (`-1`)** is a special case keyed on the *ID*, not on any role string: it is accepted (not rejected), sorts to `Player1ID`, defaults its role to `observer` when none is given, and is exempt from the major-civ check; the *other* endpoint must still be a major civ. Write one row via `store.storeTimedKnowledge('DiplomaticMessages', {...})` — a **single-row store variant that returns the new append `ID`** via `RETURNING` so the caller never re-queries (which would be racy under concurrent same-pair appends) — set `Payload` to `{}` when omitted, and pass visibility flags for the real participant(s) — both for a civ↔civ conversation, and only the real civ when the counterpart is the observer (which has no player slot). `append-message` is archival only: it does not stream, notify, run agents, enact deals, or decide whether a deal is current/accepted. Register the factory in `tools/index.ts`.
4. **Message-specific validation in `append-message`** — for `deal-proposal` and `deal-counter`, validate that `Payload.Deal` exists; `Payload.Value1` and `Payload.Value2` are optional per-item value snapshots for either ordered player — **including a human side**, whose items the VP AI (`CvDealAI`) also values, so there is no human-undefined restriction. For `deal-reject`, validate that `Payload.ProposalMessageID` references an earlier proposal/counter message between the same players; **either endpoint may speak it** — the counterparty *declines* the offer, or the original proposer *retracts* their own offer (so there is **no separate `deal-retract` type**; retraction is reuse of `deal-reject` by the proposer). The **close-conversation special message** is just an append with the close `MessageType` (no separate mechanism). `deal-accept` and `deal-enacted` are **not** emitted through this public tool: **acceptance goes through the enactment route** (`enact-agent-deal`, stage 6), which performs its own validation and idempotency check (no prior `deal-enacted` for the proposal ID) before recording `deal-accept` / `deal-enacted` via the same store path, authored by the enacting party — so the check and the record are one operation. Those records carry `Payload.ProposalMessageID` and store no legality, reasons, or DLL state. The schema `MessageType` column still includes `deal-accept` / `deal-enacted` because those rows live in the same table. (A `deal-reject` "doesn't matter" beyond being archived — it triggers no enactment and no expiration bookkeeping; superseded proposals are derived, not stored.)
5. **`mcp-server/src/tools/knowledge/read-transcript.ts`** (new, with a getter under `knowledge/getters/`) — a read tool taking `{ PlayerAID, PlayerBID, MessageType?, Role? }`, deriving the same ordered player pair, and returning messages ordered by append `ID` so both directions read as a single thread. The optional `MessageType` filters by message type (pushed to SQL) and the optional `Role` filters to messages whose **speaker** holds that free-form role (e.g. `diplomat`) — the role column exists for display and for this transcript filtering. The query filters by player pair rather than by one speaker's visibility; visibility remains useful for generic knowledge readers and future UI filtering. Register it in `tools/index.ts`.

## Reuse

`createTimedKnowledgeTable` / `createTimedKnowledgeIndexes` (`schema/setup.ts`, `schema/table-utils.ts`); `storeTimedKnowledgeBatch` (`knowledge/store.ts`); the `ToolBase` read/write shape and getter idiom modeled on `get-opinions` / `get-players`; the `toolFactories` registry in `tools/index.ts`.

## Verify

Via MCP client tool calls against a running game (no agent involvement): `append-message` several messages in both directions between two civ `playerID`s with **free-form roles** (e.g. both sides voiced as `diplomat`, or a human side as `the leader`), plus one close-conversation special message; append a deal proposal that carries **both `Payload.Value1` and `Payload.Value2`** (confirm a human-side value map is accepted, not rejected). Append a `deal-reject` referencing that proposal from the **counterparty** (a decline) and another from the **proposer themselves** (a retraction) — confirm **both** are accepted, since either endpoint may reject. Confirm `append-message` **refuses `deal-accept`** (and `deal-enacted`): acceptance is exercised through the enactment route in stage 6, not here. Append one message with the observer sentinel (`-1`) as a counterpart and confirm it is stored with the `observer` role, sorted to `Player1ID`, with visibility set only for the real civ. `read-transcript` returns messages in append-ID order as one thread whether requested as `{ PlayerAID, PlayerBID }` or `{ PlayerBID, PlayerAID }`; with `MessageType: "deal-proposal"` it returns only proposals, and with `Role: "diplomat"` only diplomat-spoken messages. Stop and restart mcp-server, re-read — the transcript and its roles persist. Confirm no thread/status table was created, no live action was triggered by append alone, and both participant visibility flags are set on every civ↔civ row.

## Done when

A conversation between two civs can be appended to and read back as one ordered, durable thread keyed only by game + the player pair ordered by `playerID`, surviving a restart — exercised purely through the new tools.
