# Stage 1 — mcp-server: durable conversation transcript store

> **✅ Done.** Verified through MCP tool calls against a running game, survives restart.
> Part of the interactive-diplomacy plan. Shared design and watch-items live in [README.md](README.md); requirements in [specs.md](specs.md).

## Objective

Conversation transcripts are **durable** in the mcp-server, surviving a restart, keyed by the game and a **player pair ordered by `playerID`**. There is exactly **one conversation per pair of major-civ players**, so the store needs no thread identity, no thread table, and no status column (specs §6) — the conversation *is* the append-ID-ordered list of messages between the two players.

The store holds **only the messages** — the two players, each player's role, the speaker, content, message type, turn, timestamp, and optional `Payload` metadata. It does **not** store LLM internals (reasoning, scratch state, tool traces), which stay transient in vox-agents. Legality is never stored; a proposal exists because it was put in the conversation, enactment re-validates live (stage 6), and `deal-enacted` records successful orchestration rather than DLL state.

## What was built

- **Schema** ([schema/timed.ts](../../../mcp-server/src/knowledge/schema/timed.ts)) — a `DiplomaticMessage` interface extending `TimedKnowledge` with `Player1ID` / `Player2ID` (`min` / `max` `playerID`), `Player1Role` / `Player2Role` (**free-form role descriptors** following the EnvoyThread tradition — an agent name such as `diplomat` / `negotiator`, a human's `UserIdentity.role` such as `the leader`, or `observer`; they do **not** encode human-vs-LLM), `SpeakerID`, `MessageType` (`text`, `close`, `deal-proposal`, `deal-counter`, `deal-accept`, `deal-reject`, `deal-enacted`), and `Content`. The inherited `Payload` JSON carries optional metadata: proposal/counter messages store terms at `Payload.Deal` (with optional `rationale` / one-sentence `message` — pinned in stage 3) and proposal-time per-item values at `Payload.Value1` / `Payload.Value2`; accept/reject/enacted messages store `Payload.ProposalMessageID`. Exported from `schema/index.ts`; `DiplomaticMessages` added to the `KnowledgeDatabase` interface in `schema/base.ts`.
- **Table** ([schema/setup.ts](../../../mcp-server/src/knowledge/schema/setup.ts)) — created in `setupKnowledgeDatabase()` via `createTimedKnowledgeTable(db, 'DiplomaticMessages')` with the standard timed indexes plus a **player-pair index** on `(Player1ID, Player2ID, ID)` for transcript reads. `TimedKnowledge`, not `MutableKnowledge` — no status columns. `Turn` is metadata; append `ID` is canonical transcript order.
- **`append-message`** ([tools/actions/append-message.ts](../../../mcp-server/src/tools/actions/append-message.ts)) — a non-read-only `ToolBase` write taking endpoint-oriented input `{ PlayerAID, PlayerBID, PlayerARole?, PlayerBRole?, SpeakerID, MessageType, Content, Payload?, Turn? }`. It orders the two player IDs server-side, remaps the free-form roles to `Player1Role` / `Player2Role`, defaults `Turn` to the current server turn (`knowledgeManager.getTurn()`) when omitted, and validates that the endpoints are distinct, that `SpeakerID` is one of them, and that each non-observer endpoint is a **major civilization** (checked against cached `PlayerInformations`, not a live per-append query). The **observer sentinel `-1`** is a special case keyed on the *ID*: accepted (not rejected), sorted to `Player1ID`, defaulted to the `observer` role, exempt from the major-civ check; the other endpoint must still be a major civ. It writes one row via a single-row store variant returning the new append `ID` (so the caller never re-queries, which would be racy under concurrent same-pair appends), sets `Payload` to `{}` when omitted, and sets visibility flags for the real participant(s). It is **archival only** — it does not stream, notify, run agents, or enact deals. Registered in `tools/index.ts`.
- **Message-specific validation in `append-message`** — `deal-proposal` / `deal-counter` require `Payload.Deal` (the optional `Payload.Value1` / `Payload.Value2` snapshots may be present for **either** ordered player, including a human side, whose items the VP AI also values). `deal-reject` requires a `Payload.ProposalMessageID` referencing an earlier proposal/counter between the same players and **may be spoken by either endpoint** — the counterparty *declines* or the original proposer *retracts* (so there is no separate `deal-retract` type). The close special message is just an append with the `close` `MessageType`. `deal-accept` and `deal-enacted` are **refused** here: they are written only by the enactment route (`enact-agent-deal`, stage 6), which performs its own idempotency check and records both via the same store path, so the check and the record are one operation.
- **`read-transcript`** ([tools/knowledge/read-transcript.ts](../../../mcp-server/src/tools/knowledge/read-transcript.ts), with a getter under `knowledge/getters/`) — takes `{ PlayerAID, PlayerBID, MessageType?, Role? }`, derives the same ordered pair, and returns messages ordered by append `ID` so both directions read as one thread. `MessageType` filters in SQL; `Role` filters to messages whose **speaker** holds that free-form role. The query filters by player pair, not by a single speaker's visibility. Registered in `tools/index.ts`.

## Reuse

`createTimedKnowledgeTable` / `createTimedKnowledgeIndexes` ([schema/setup.ts](../../../mcp-server/src/knowledge/schema/setup.ts), `schema/table-utils.ts`); the single-row variant of `storeTimedKnowledge` (`knowledge/store.ts`); the `ToolBase` read/write shape and getter idiom modeled on `get-opinions` / `get-players`; the `toolFactories` registry in `tools/index.ts`.

## Verify

Via MCP client tool calls against a running game (no agent involvement): `append-message` several messages in both directions between two civ `playerID`s with **free-form roles**, plus one close message; append a proposal carrying **both `Payload.Value1` and `Payload.Value2`** (a human-side value map is accepted, not rejected); append a `deal-reject` from the **counterparty** (a decline) and another from the **proposer** (a retraction) — both accepted; confirm `append-message` **refuses `deal-accept`** and `deal-enacted`; append one message with the observer sentinel `-1` and confirm it is stored with the `observer` role, sorted to `Player1ID`, visible only to the real civ. `read-transcript` returns append-ID order as one thread whether requested as `{ PlayerAID, PlayerBID }` or reversed; `MessageType: "deal-proposal"` returns only proposals and `Role: "diplomat"` only diplomat-spoken messages. Restart mcp-server and re-read — transcript and roles persist. No thread/status table exists, append alone triggers no live action, and both participant visibility flags are set on every civ↔civ row.

## Done when

A conversation between two civs can be appended to and read back as one ordered, durable thread keyed only by game + the player pair ordered by `playerID`, surviving a restart — exercised purely through the new tools.
