# Refactor live envoys to speak via a `send-message` tool

> Part 1 of the diplomat refactoring. Self-contained and shippable on its own.
> Part 2 (cache-aware envoy prompt structure) is a separate follow-up: `05.2-cache-aware-envoy-prompt.md`.

## Context

Today the live envoys (Diplomat, Spokesperson) speak to the counterpart by emitting **raw assistant free text**, while every action (negotiation, briefings, closing) goes through a tool. The model must constantly decide "do I write text now, or call a tool?" — and it gets this wrong (narrating instead of acting, or calling a tool when it should just reply). The free-text/tool duality is the root of the confusion.

This refactor makes **speaking a tool too**: a single `send-message` tool whose `Message` argument is streamed to the client exactly like free text is today. Once "speak" is an explicit action the model's action space is uniform — it always picks a tool — which removes the ambiguity.

**Why this actually works on the deployed model.** The configured model is `openai-compatible/MiniMax-M2.7` (`config.json`), not Anthropic. The `required`→`auto` rewrite in `vox-context.ts` (~line 773) only fires for `provider === "anthropic"`, so on the deployed model `toolChoice: "required"` **is honored** — the model is forced to call a tool every step, so making "speak" a tool genuinely unifies the action space. On Anthropic the rewrite neutralizes the force, so we keep a **free-text fallback** (see below); the change is safe regardless of provider.

> **Streaming is load-bearing, not polish.** Because `required` is honored on the deployed model, *every* normal reply now flows through `send-message`. So streaming the `Message` argument back as text is what preserves today's token-by-token bubble for all replies — without it, every diplomat reply would regress to appearing all at once. The streamer/decoder below is therefore a required part of this plan, not an optimization.

Decisions confirmed with the user:
- **Scope: all live envoys.** Diplomat *and* Spokesperson speak only through `send-message`. Shared mechanics live on `LiveEnvoy`.
- **Terminal tools.** `send-message`, `call-negotiator`, and `close-conversation` are all terminal for the Diplomat (Spokesperson: just `send-message`). The model may call `send-message` alone, `call-negotiator` alone, or both in one step — any ends the turn.
- **Deal-gating applies only to a counterpart-authored open proposal.** When *our* side has a pending proposal the ball is on the other side, so no restriction.

## Key constraint

`vox-context.ts` (~line 773) rewrites `toolChoice: "required"` → `"auto"` for the **anthropic** provider, so we cannot hard-force tool calls there. The prompt steers the model, and we keep a **free-text fallback**: if the model emits raw text instead of `send-message`, that text is still streamed natively and archived as the reply.

## Streaming design (the one intricate part — isolate and unit-test it)

The AI SDK (v6.0.174) delivers `tool-input-start` / `tool-input-delta` / `tool-call` / `tool-result` chunks to `streamText`'s `onChunk`. `tool-input-delta` carries only raw `inputTextDelta` (no pre-parsed partial — confirmed in the SDK types), so turning the `send-message` argument back into clean streamed text needs a small partial-JSON decode. All of this lives in **one new, pure, well-tested module** so the complexity is contained.

The web route runs every chunk through a per-request **send-message streamer**:
- `tool-input-start` (toolName `send-message`) → open a text stream keyed by `chunk.id`.
- `tool-input-delta` for that id → accumulate raw JSON, decode the partial `Message` string value so far, emit the **new suffix** as a normal `text-delta` SSE event (`{ type: 'text-delta', text, id }`), tracking how much was already emitted.
- `tool-call` (toolName `send-message`) → authoritative full input; emit any remaining suffix (covers providers that stream the arg whole, e.g. Gemini), then close the stream.
- `tool-call` / `tool-result` for `send-message` are **swallowed** (not forwarded), so the UI renders the message as a text bubble, never a tool-call card.
- All other chunks (native `text-delta` greetings/fallback, other tools' `tool-call`/`tool-result`) pass through unchanged. We deliberately do **not** suppress native free text when a `send-message` also appears in the same turn — both are shown to the user, and archival mirrors the display (it persists text and `send-message` in their streamed order; see *Reply extraction*), so the live view and the reloaded transcript match.

Details that keep it safe:
- **Partial-JSON decode is synchronous** via a hand-rolled `decodeJsonStringField(raw, "Message")` that walks the JSON string honoring escapes and backs off at an incomplete escape / `\u` at the buffer boundary. We deliberately avoid the SDK's `parsePartialJson` because it is `async` and `concurrency.ts` does **not** await `onChunk` (confirmed: `onChunk` is invoked synchronously) — async work there would race chunk ordering. The schema's single `Message` string field makes the field unambiguous/first.
- `concurrency.ts` prefixes streaming chunk `id`s with a per-call counter; `tool-input-start`/`tool-input-delta` both carry `id`, so they correlate consistently. The streamer tracks at most one open `send-message` stream per turn, so it does not need to reconcile the delta `id` against the `tool-call`'s `toolCallId`.
- The nested negotiator runs with **no stream callback** (`agent-tools.ts` calls `context.execute(targetName, agentInput)` with no callback — confirmed), so its tokens never reach this streamer.
- **Greeting/special mode also speaks through `send-message`.** With `toolChoice: "required"` honored on the deployed model, special mode cannot use an empty tool set; `LiveEnvoy.prepareStep` sets `activeTools = ["send-message"]` (not `[]`), so the greeting is a `send-message` call streamed as `text-delta` like any other reply — one path for all spoken output.

## Deal-gating (authoritative, no extra state)

When a deal authored by the **counterpart** is open on the table, the Diplomat is restricted to two tools (`call-negotiator`, `send-message`) so it can't wander into briefings/analyst calls when the ball is in its court.

Derive the gate from the **authoritative durable reduction** — the same source the deal context uses — **not** the in-memory cache. The cache (`input.messages[].deal`) is reconciled best-effort, so after a disconnect on an accept/reject it can still read `open` and wrongly keep restricting the next turn. `Diplomat.prepareStep` therefore reads `readActiveProposal(player1ID, player2ID)` (the exact `DealReduction` the negotiator and the accept/reject routes already use) and gates on it:

```ts
// Diplomat.prepareStep
if (input.diplomacy && !this.isSpecialMode(input)) {
  const reduction = await readActiveProposal(input.player1ID, input.player2ID);
  if (counterpartOpenProposal(reduction, input.agent)) {
    config.activeTools = ["call-negotiator", "send-message"];
  }
}
```

No module-level state, no `WeakMap`, no flag threaded out of `getInitialMessages`, and `diplomat-utils.ts` / `buildDealContextMessage` stay **untouched**. The gate reads the durable store where it's used, so it can never restrict on a stale cache. Because `call-negotiator` is terminal, the gate is constant across a turn, so the only cost is a repeat `readActiveProposal` on the uncommon multi-step turn — a cheap local read. (If that ever shows up in a profile, 05.2's deal-state refactor can memoize it per run.)

## Stop logic (shared, with a hard step ceiling)

Move the diplomat's bespoke stop logic up to `LiveEnvoy`, generalized over a `getCompletionTools()` set. Because `send-message` is itself a completion tool, "the model spoke" collapses into "the model called a completion tool" — the free-text branch survives only as the Anthropic fallback.

All live envoys get a **hard step ceiling of 10** (`LiveEnvoy.maxSteps = 10`, overriding the base default of 3). This matters because the diplomat's bespoke check today has *no* ceiling, and moving it shared would also strip the base `VoxAgent` `maxSteps` backstop that the Spokesperson currently relies on — so a runaway support-tool loop (`get-briefing` forever) or a string of empty responses could otherwise spin without end. The ceiling is checked **before** the "keep working" branch so a stuck loop always terminates:

1. Persist via `super.stopCheck` (Envoy) — every checked step is still archived to the thread.
2. **Stop** if any step called a completion tool (`send-message` / `call-negotiator` / `close-conversation`).
3. **Stop** once `allSteps.length >= this.maxSteps` (10) — the hard ceiling, checked before the support-tool branch.
4. **Continue** if the last step has a pending non-completion (support) tool (`get-briefing`, `get-diplomatic-events`, `call-diplomatic-analyst`) — the envoy means to keep working.
5. Otherwise **stop** once any step produced spoken free text (the Anthropic fallback).

When the loop ends with **no spoken reply** (the ceiling was hit, or the model produced nothing usable), the envoy must still answer the counterpart. A shared `retryMessage` constant (e.g. _"My apologies — I lost my train of thought. Could you say that again?"_) is then both streamed to the client and archived as the turn's reply (see *Reply extraction* and *Web route*), so a stuck turn degrades into a polite retry request instead of dead air.

## Changes by file

### New files
- **`vox-agents/src/envoy/send-message-tool.ts`** — `createSendMessageTool(context)` via `createSimpleTool`. Schema `{ Message: z.string().describe(...) }`. `execute` returns a short confirmation (e.g. `"Message delivered."`); it does not mutate the thread (streaming + archival are handled elsewhere). Export the tool name as a camelCase constant `sendMessageToolName = "send-message"` (the repo's exported-constant convention — matches `worldContext` / `communicationStyle` rather than `SCREAMING_CASE`).
- **`vox-agents/src/web/routes/send-message-stream.ts`** — `createSendMessageStreamer(emitTextDelta)` returning `{ handleChunk(chunk): boolean }` (true = swallowed), plus the pure exported `decodeJsonStringField(raw, field)` for unit testing.

### Envoy core
- **`vox-agents/src/envoy/live-envoy.ts`**
  - `toolChoice = "required"` (honored on the deployed model; neutralized to `auto` on Anthropic).
  - `maxSteps = 10` (override the base default of 3) — the shared hard ceiling for every live envoy.
  - `getActiveTools` base list adds `"send-message"`; `getExtraTools` registers `createSendMessageTool(context)` alongside `get-briefing`.
  - Add `protected getCompletionTools(): Set<string>` returning `{"send-message"}` by default.
  - Move the **shared stop logic** here (the five-step rule above, including the `maxSteps` ceiling), generalized over `getCompletionTools()`.
  - `prepareStep`: special mode sets `activeTools = ["send-message"]` (greetings speak via the tool too).
- **`vox-agents/src/envoy/envoy.ts`** — `convertToModelMessages`: in the tool-role branch, **skip `send-message` tool-result parts** (drop the "Message delivered." confirmation; if that empties the tool message, `continue` as today). Do **not** rewrite anything — `send-message` tool-call parts stay tool-call parts, so the model's own history consistently shows it spoke by calling `send-message`.

### Agents
- **`vox-agents/src/envoy/diplomat.ts`**
  - `getActiveTools` adds `"send-message"`.
  - Override `getCompletionTools()` → `{"send-message","call-negotiator","close-conversation"}`. Delete the now-redundant `completionTools` static and the bespoke `stopCheck` (logic now shared in `LiveEnvoy`).
  - `prepareStep`: keep the non-diplomacy tool gating; add the counterpart-open-proposal restriction read from the authoritative `readActiveProposal` reduction (above). `getInitialMessages` is unchanged.
  - `getSystem`: rewrite resources/expectations — speak to the counterpart **only** by calling `send-message`; describe the negotiator handoff; when a counterpart deal is on the table, either hand it to the negotiator or reply with `send-message`. Update `getDefaultAddon` to mention `send-message`.
- **`vox-agents/src/envoy/spokesperson.ts`** — `getActiveTools` adds `"send-message"`; system prompt updated to say it answers by calling `send-message` (replacing the "do not send out in a text block" guidance).

### Deal-state predicate (shared)
- **`vox-agents/src/utils/diplomacy/deal-reduce.ts`** — add `counterpartOpenProposal(reduction, agentSeat)` → `reduction.status === "open" && !!reduction.active && reduction.active.SpeakerID !== agentSeat`. (The companion `ownPendingProposal` and the negotiator's inline-check cleanup are deferred to `05.2`.)

### Reply extraction / archival
- **`vox-agents/src/utils/diplomacy/transcript-utils.ts`** — add `collectSpokenReply(messages)` that captures **exactly what was displayed**: walk the assistant messages in order and, within each message, walk the content parts in their original order, concatenating every `text` part's text and every `send-message` tool-call part's `Message` input. That is the same sequence the client rendered (passed-through native `text-delta`s interleaved with the `send-message` text the streamer converted), so a reload reproduces the live view — closing the leak where a model that narrates *and then* calls `send-message` would show both live but persist only the tool text. Returns `""` when nothing was spoken. Also export a shared `retryMessage` constant here (used by the commit path and the web route). Keep `joinAssistantText` as-is for its other callers. Re-export both from `transcript.ts`.
- **`vox-agents/src/utils/diplomacy/chat-turn-commit.ts`** — replace the **single** `joinAssistantText(...)` reply-capture call (`chat-turn-commit.ts:129`) with `collectSpokenReply(...) || retryMessage`, so a turn that hit the step ceiling with nothing spoken still archives the retry line. *(There is exactly one reply-capture site; deal auto-responses already flow through this same unified path.)*

### Web route
- **`vox-agents/src/web/routes/agent.ts`** — in the `POST /agents/message` handler, build `createSendMessageStreamer((text, id) => sendEvent('message', { type:'text-delta', text, id }))` and route `streamCallback.OnChunk` through it: `if (!streamer.handleChunk(chunk)) sendEvent('message', chunk)`. Track whether any spoken text reached the client this turn (a native `text-delta` passthrough or a `send-message` delta); if none did, after the run emit `retryMessage` as a final `text-delta` — the same line `chat-turn-commit` archives — so a ceiling-stop is visible to the user, not silent.

### UI
- No change required: `send-message` is delivered as `text-delta` (already handled by `useThreadMessages.ts`), its tool chunks are swallowed server-side, and on reload the durable store returns clean `text` rows. (Leave the unrelated uncommitted edits in `useThreadMessages.ts` / `ChatDetailView.vue` alone.)
