## Context

Today the live envoys (Diplomat, Spokesperson) speak to the counterpart by emitting **raw assistant free text**, while every action (negotiation, briefings, closing) goes through a tool. The model must constantly decide "do I write text now, or call a tool?" тАФ and it gets this wrong (narrating instead of acting, or calling a tool when it should just reply). The free-text/tool duality is the root of the confusion.

This refactor makes **speaking a tool too**: a single `send-message` tool whose `Message` argument is streamed to the client exactly like free text is today. Once "speak" is an explicit action the model's action space is uniform тАФ it always picks a tool тАФ which removes the ambiguity.

**Why this actually works on the deployed model.** The configured model is `openai-compatible/MiniMax-M2.7` (`config.json`), not Anthropic. The `required`тЖТ`auto` rewrite in `vox-context.ts` (~line 773) only fires for `provider === "anthropic"`, so on the deployed model `toolChoice: "required"` **is honored** тАФ the model is forced to call a tool every step, so making "speak" a tool genuinely unifies the action space. On Anthropic the rewrite neutralizes the force, so we keep a **free-text fallback** (see below); the change is safe regardless of provider.

> **Streaming is load-bearing, not polish.** Because `required` is honored on the deployed model, *every* normal reply now flows through `send-message`. So streaming the `Message` argument back as text is what preserves today's token-by-token bubble for all replies тАФ without it, every diplomat reply would regress to appearing all at once. The streamer/decoder below is therefore a required part of this plan, not an optimization.

Decisions confirmed with the user:
- **Scope: all live envoys.** Diplomat *and* Spokesperson speak only through `send-message`. Shared mechanics live on `LiveEnvoy`.
- **Terminal tools.** `send-message`, `call-negotiator`, and `close-conversation` are all terminal for the Diplomat (Spokesperson: just `send-message`). The model may call `send-message` alone, `call-negotiator` alone, or both in one step тАФ any ends the turn.
- **Deal-gating applies only to a counterpart-authored open proposal.** When *our* side has a pending proposal the ball is on the other side, so no restriction.

## Key constraint

`vox-context.ts` (~line 773) rewrites `toolChoice: "required"` тЖТ `"auto"` for the **anthropic** provider, so we cannot hard-force tool calls there. The prompt steers the model, and we keep a **free-text fallback**: if the model emits raw text instead of `send-message`, that text is still streamed natively and archived as the reply.

## Streaming design (the one intricate part тАФ isolate and unit-test it)

The AI SDK (v6.0.174) delivers `tool-input-start` / `tool-input-delta` / `tool-call` / `tool-result` chunks to `streamText`'s `onChunk`. The chunk `onChunk` receives is a `TextStreamPart`, whose `tool-input-delta` variant carries only a raw JSON fragment in its **`delta`** field (no pre-parsed partial), so turning the `send-message` argument back into clean streamed text needs a small partial-JSON decode. (The `inputTextDelta` field lives only on the lower-level UI/provider stream-part types `onChunk` does not deliver, so the streamer reads `chunk.delta`.) All of this lives in **one new, pure, well-tested module** (`utils/models/send-message-stream.ts`, a model-stream transform sibling to `concurrency.ts`; the web route owns only the SSE wiring) so the complexity is contained.

The web route runs every chunk through a per-request **send-message streamer**:
- `tool-input-start` (toolName `send-message`) тЖТ open a text stream keyed by `chunk.id`.
- `tool-input-delta` for that id тЖТ append `chunk.delta` to the accumulated raw JSON, decode the partial `Message` string value so far, emit the **new suffix** as a normal `text-delta` SSE event (`{ type: 'text-delta', text, id }`), tracking how much was already emitted.
- `tool-call` (toolName `send-message`) тЖТ authoritative full input (`chunk.input.Message`); resolve which delta stream it closes (see id-correlation below), emit any remaining suffix under that stream's id, then drop the stream. When no delta stream matches (a provider that streamed the arg whole, e.g. Gemini), emit the full `Message` as a fresh bubble keyed on `chunk.toolCallId`.
- `tool-call` / `tool-result` for `send-message` are **swallowed** (not forwarded), so the UI renders the message as a text bubble, never a tool-call card.
- All other chunks (native `text-delta` greetings/fallback, other tools' `tool-call`/`tool-result`) pass through unchanged. We deliberately do **not** suppress native free text when a `send-message` also appears in the same turn тАФ both are shown to the user, and archival mirrors the display (it persists text and `send-message` in their streamed order; see *Reply extraction*), so the live view and the reloaded transcript match.

Details that keep it safe:
- **Partial-JSON decode is synchronous** via a hand-rolled `decodeJsonStringField(raw, "Message")` that walks the JSON string honoring escapes and backs off at an incomplete escape / `\u` at the buffer boundary. We deliberately avoid the SDK's `parsePartialJson` because it is `async` and `concurrency.ts` does **not** await `onChunk` (confirmed: `onChunk` is invoked synchronously) тАФ async work there would race chunk ordering. The schema's single `Message` string field makes the field unambiguous/first.
- **Per-call id tracking (not one stream per turn).** The model may emit more than one `send-message` call in a single step (terminal handling runs only *after* the step), so the streamer keeps a `Map` of in-flight streams keyed by the chunk `id`; two concurrent calls never share a buffer or cross-assign suffixes. `concurrency.ts` prefixes streaming chunk `id`s with a per-call counter, and `tool-input-start`/`tool-input-delta` both carry `id`, so a stream's deltas correlate consistently. The `tool-call`/`tool-result` chunks instead carry the **unprefixed** `toolCallId` (they have no `id`), so on `tool-call` the streamer matches it back to the right stream by an exact hit then a `-<toolCallId>` suffix hit (the `-` boundary prevents a spurious tail match). `send-message` is built via `createSimpleTool`/`dynamicTool`, so its tool chunks are dynamic but still carry `toolName`, which is what the streamer keys the swallow on.
- The `onChunk` callback only ever delivers `text-delta` / `reasoning-delta` / `source` / `tool-call` / `tool-input-start` / `tool-input-delta` / `tool-result` / `raw` (confirmed in the SDK's `StreamTextOnChunkCallback`), so there is no `tool-input-end` to handle; the streamer's default branch passes every non-`send-message` chunk through.
- The nested negotiator runs with **no stream callback** (`agent-tools.ts` calls `context.execute(targetName, agentInput)` with no callback тАФ confirmed), so its tokens never reach this streamer.
- **Greeting/special mode also speaks through `send-message`.** With `toolChoice: "required"` honored on the deployed model, special mode cannot use an empty tool set; `LiveEnvoy.prepareStep` sets `activeTools = ["send-message"]` (not `[]`), so the greeting is a `send-message` call streamed as `text-delta` like any other reply тАФ one path for all spoken output.

## Deal-gating (authoritative, no extra state)

When a deal authored by the **counterpart** is open on the table, the Diplomat is restricted to two tools (`call-negotiator`, `send-message`) so it can't wander into briefings/analyst calls when the ball is in its court.

Derive the gate from the **authoritative durable reduction** тАФ the same source the deal context uses тАФ **not** the in-memory cache. The cache (`input.messages[].deal`) is reconciled best-effort, so after a disconnect on an accept/reject it can still read `open` and wrongly keep restricting the next turn. `Diplomat.prepareStep` therefore reads `readActiveProposal(player1ID, player2ID)` (the exact `DealReduction` the negotiator and the accept/reject routes already use) and gates on it:

```ts
// Diplomat.prepareStep
if (input.diplomacy && !this.isSpecialMode(input)) {
  const reduction = await readActiveProposal(input.player1ID, input.player2ID);
  if (counterpartOpenProposal(reduction, input.agent)) {
    config.activeTools = ["call-negotiator", "send-message"];
  }
}
```

No module-level state, no `WeakMap`, no flag threaded out of `getInitialMessages`, and `diplomat-utils.ts` / `buildDealContextMessage` stay **untouched**. The gate reads the durable store where it's used, so it can never restrict on a stale cache. Because `call-negotiator` is terminal, the gate is constant across a turn, so the only cost is a repeat `readActiveProposal` on the uncommon multi-step turn тАФ a cheap local read. (If that ever shows up in a profile, 05.2's deal-state refactor can memoize it per run.)

## Stop logic (shared, with a hard step ceiling)

Move the diplomat's bespoke stop logic up to `LiveEnvoy`, generalized over a `getCompletionTools()` set. Because `send-message` is itself a completion tool, "the model spoke" collapses into "the model called a completion tool" тАФ the free-text branch survives only as the Anthropic fallback.

All live envoys get a **hard step ceiling of 10** (`LiveEnvoy.maxSteps = 10`, overriding the base default of 3). This matters because the diplomat's bespoke check today has *no* ceiling, and moving it shared would also strip the base `VoxAgent` `maxSteps` backstop that the Spokesperson currently relies on тАФ so a runaway support-tool loop (`get-briefing` forever) or a string of empty responses could otherwise spin without end. The ceiling is checked **before** the "keep working" branch so a stuck loop always terminates:

1. Persist via `super.stopCheck` (Envoy) тАФ every checked step is still archived to the thread.
2. **Stop** if any step called a completion tool (`send-message` / `call-negotiator` / `close-conversation`).
3. **Stop** once `allSteps.length >= this.maxSteps` (10) тАФ the hard ceiling, checked before the support-tool branch.
4. **Continue** if the last step has a pending non-completion (support) tool (`get-briefing`, `get-diplomatic-events`, `call-diplomatic-analyst`) тАФ the envoy means to keep working.
5. Otherwise **stop** once any step produced spoken free text (the Anthropic fallback).

When the loop ends with **no spoken reply** (the ceiling was hit, or the model produced nothing usable), the envoy must still answer the counterpart. A shared `retryMessage` constant (e.g. _"My apologies тАФ I lost my train of thought. Could you say that again?"_) is then both streamed to the client and archived as the turn's reply (see *Reply extraction* and *Web route*), so a stuck turn degrades into a polite retry request instead of dead air.
