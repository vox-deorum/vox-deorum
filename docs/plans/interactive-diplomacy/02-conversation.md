### Deviations / decisions

- **Store-aligned `EnvoyThread` (ordered player1/2), not A/B.** The thread mirrors the `DiplomaticMessage` schema (player1/2 ordered by id, free-form roles); the *voiced* seat is derived (`thread.agent` = its playerID) rather than positional, and the audience is the other endpoint. This replaced an earlier A/B caller/voiced draft.
- **`agent` is a playerID, and identity comes from parameters.** The executable agent name is the agent seat's role (`roleOf(thread, thread.agent)`); the voiced civ/leader is derived from the typed parameters through abstract `Envoy` helpers, never stored. UI labels are served via response **enrichment** (`voicedCiv`/`audienceCiv`/`voicedID` on `Create/GetChatResponse`) so the thread stays schema-pure.
- **`SpeakerID` is per-message, not thread state** ŌĆö resolved at send time (audience for a caller's text / Web close; the agent seat for the diplomat's reply / close-from-tool).
- **Diplomacy-only write-through** (agreed with the operator): observer/telepathist chats adopt the ordered-pair shape but are *not* persisted to the store this stage. Work-item 1's "observer chats persist through the same store" is therefore deferred (the store already supports it via the `-1` sentinel; only the route wiring is deferred).
- **`GET /chat/:id` does not re-hydrate after a restart**; reopening a diplomacy conversation goes through `AgentSelectDialog` ŌåÆ `POST /chat`, which hydrates from the store (the id is deterministic). This matches the Verify flow ("reopen the conversation").

## Objective

A human seated as a major civilization can, **from the Web**, open a free-text conversation with an LLM-played civ and exchange messages; the LLM defaults to the voice of **that seat's configured diplomat agent** (specs ┬¦1, ┬¦7), with an explicit local-operator override available for debugging. The transcript persists through the stage-1 store and survives a restart. Closing the conversation writes a special message and **locks it for the rest of the current turn** (specs ┬¦8). No deals yet ŌĆö this is the conversation MVP, the easiest direction (humanŌåÆLLM) built end to end.

## Approach

Diplomacy is inherently civ-to-civ and **direction-agnostic** (specs ┬¦2, ┬¦5): a conversation is keyed by the player pair ordered by `playerID` plus the current speaker/recipient, either of which may be human or LLM. Today's chat API is "client names an agent"; for diplomacy the Web client names the target civ and the server defaults to that seat's configured diplomat while allowing a local debug override. To make the in-memory thread map 1:1 onto the stage-1 transcript rows, **align `EnvoyThread`'s identity with the mcp-server's message shape** ŌĆö the player pair ordered by `playerID` (`player1`/`player2`) with free-form per-player roles, plus an `agent` field naming the agent-voiced seat ŌĆö replacing the single `playerID` + free-form `userIdentity`. The current speaker is per-message, not thread state; participant identity is derived from the agent's typed parameters, not stored. Do this as one coordinated refactor across `EnvoyThread`, chat routes, endpoint labeling, context/parameter resolution, and `getPlayerAssignments`, so diplomacy and ordinary envoy/telepathist chat share one endpoint-pair model. The in-memory `chatSessions` map stops being the source of truth and becomes a **write-through cache** over the stage-1 transcript store.

## Verify

End to end on the Web: launch a session with an LLM civ whose seat config names a diplomat. A human seated as another civ opens a conversation with it and exchanges free-text; the LLM replies in the configured diplomat's voice (and uses the configured model), unless the local operator selected an override. Stop and restart; reopen the conversation ŌĆö the full transcript is still there (read from the store, not memory). Close the conversation; confirm it cannot be reopened on the same turn but can on a later turn.
