# Stage 2 — vox-agents + Web: human↔LLM civ-to-civ conversation (CORE MVP)

> **✅ Done.** Backend + UI type-check clean; full vox-agents unit suite green with new pure-helper + Envoy-identity tests. End-to-end (LLM reply, restart-survival, same-turn lock) verified against a running game with an LLM-diplomat seat.
> Part of the interactive-diplomacy plan. Shared design and watch-items live in [README.md](README.md); requirements in [specs.md](specs.md).

## Objective

A human seated as a major civilization can, **from the Web**, open a free-text conversation with an LLM-played civ and exchange messages; the LLM defaults to the voice of **that seat's configured diplomat agent** (specs §1, §7), with an explicit local-operator override available for debugging. The transcript persists through the stage-1 store and survives a restart. Closing the conversation writes a special message and **locks it for the rest of the current turn** (specs §8). No deals yet — this is the conversation MVP, the easiest direction (human→LLM) built end to end.

Diplomacy is inherently civ-to-civ and direction-agnostic (specs §2, §5): a conversation is keyed by the player pair ordered by `playerID`, either of which may be human or LLM. To make the in-memory thread map 1:1 onto the stage-1 transcript rows, `EnvoyThread`'s identity is aligned with the mcp-server message shape, and the `chatSessions` map becomes a **write-through cache** over the store rather than the source of truth.

## What was built

- **Store-aligned `EnvoyThread`** ([chat.ts](../../../vox-agents/src/types/chat.ts)) — the player pair is stored **ordered by `playerID`** (`player1ID` / `player2ID` = min/max + free-form `player1Role` / `player2Role`), mirroring the `DiplomaticMessage` row 1:1. `agent` is the **playerID of the agent-voiced seat**; that seat's role descriptor *is* the executable agent name (the stage-1 pinned contract), so the agent name is `roleOf(thread, thread.agent)` and the audience is the other endpoint — positions carry no caller/voiced or human/LLM meaning. `SpeakerID` is per-message (resolved at send time), not thread state. Identity (civ/leader) is **not** stored: it is derived from the typed parameters via abstract `Envoy` helpers `getParticipantIdentity` / `getSelfIdentity` ([envoy.ts](../../../vox-agents/src/envoy/envoy.ts)), overridden by `LiveEnvoy` (from `metadata.YouAre` / game state) and `Telepathist` (from `civilizationName` / `leaderName`). The old `playerID` / `userIdentity` / `civilizationName` fields were removed and consumers migrated. UI labels are served via response **enrichment** (`voicedCiv` / `audienceCiv` / `voicedID` on the chat responses) so the thread stays schema-pure.
- **Pure derivation helpers** ([transcript-utils.ts](../../../vox-agents/src/utils/diplomacy/transcript-utils.ts), unit-tested) — `roleOf` / `voicedID` / `agentName` / `audienceID` / `orderPair`; the I/O wrappers (`readTranscript` / `appendTranscriptMessage` / `appendCloseMessage`) live in [transcript.ts](../../../vox-agents/src/utils/diplomacy/transcript.ts).
- **`PlayerConfig.diplomat` / `negotiator`** ([config.ts](../../../vox-agents/src/types/config.ts)) — selected the same way as `strategist`, with their own model overrides via the per-agent `llms` map. `negotiator` is declared but unused until stage 5. `getPlayerAssignments` ([strategist-session.ts](../../../vox-agents/src/strategist/strategist-session.ts)) reports the resolved `diplomat` / `negotiator` names + models; the diplomat defaults to the built-in `diplomat` agent so the conversation route always has a voice.
- **Diplomacy routes** ([web/routes/agent.ts](../../../vox-agents/src/web/routes/agent.ts)) — `/api/agents/chat` gained a `mode: 'diplomacy'` branch: deterministic id `dipl:${gameID}:${min}:${max}`, target diplomat resolved server-side as the default voice (operator `agentName` override honored), initiator defaulting to the human-control seat, voiced in the **target** seat's context (`${gameID}-player-${targetID}`) so `YouAre` matches the civ, hydrated from `read-transcript`. `/api/agents/message` appends the caller's text via `append-message`, runs the diplomat in the target seat's context, streams the reply over SSE, then appends the reply — special tokens (`{{{Greeting}}}`) are never archived. `POST /api/agents/chat/:chatId/close` mirrors the diplomat's close path. `chatSessions` is a write-through cache for diplomacy threads.
- **`close-conversation` tool** ([close-conversation-tool.ts](../../../vox-agents/src/envoy/close-conversation-tool.ts)) on the `Diplomat`, gated to civ↔civ threads via `prepareStep`; it writes the `close` message through the shared transcript helper. Open/closed status and the same-turn resume lock are derived from the latest `close` message's turn — there is no status flag.
- **Web controls** ([ChatDetailView.vue](../../../vox-agents/ui/src/views/ChatDetailView.vue)) — a Close button, an open/closed indicator (closed-this-turn vs. reopenable), and a locked input when closed this turn, all routed through the same `append-message` close path (`api.closeAgentChat`).

**Scope note — diplomacy-only write-through.** Observer/telepathist chats adopt the ordered-pair shape but are **not yet persisted** to the store; they keep their in-memory behavior. The store already supports the observer endpoint via the `-1` sentinel (stage 1) — only the route wiring for non-diplomacy chats is deferred. Reopening a diplomacy conversation after a restart goes through `AgentSelectDialog` → `POST /chat`, which re-hydrates from the store on the deterministic id (`GET /chat/:id` does not re-hydrate).

## Reuse

`Diplomat` / `LiveEnvoy` / `Envoy` (`src/envoy/*.ts`) and their tools (`get-briefing`, `get-diplomatic-events`); the chat routes + SSE streaming and `chatSessions` map; the per-seat agent-selection idiom used for strategists; `getPlayerAssignments`; the human-control seat resolver; the Vue chat surface (`ui/src/components/chat/*`, `ChatDetailView.vue`) and `ui/src/api/client.ts`.

## Verify

End to end on the Web: launch a session with an LLM civ whose seat config names a diplomat. A human seated as another civ opens a conversation and exchanges free-text; the LLM replies in the configured diplomat's voice and model, unless the operator selected an override. Restart, reopen the conversation — the full transcript is still there (read from the store, not memory). Close the conversation; confirm it cannot be reopened on the same turn but can on a later turn.

## Done when

A human can hold a durable, restart-surviving, civ-to-civ free-text conversation with an LLM-played civ on the Web, voiced by the target seat's configured diplomat by default, with working close-and-lock semantics — and the transcript is the mcp-server store, with `chatSessions` reduced to a write-through cache.
