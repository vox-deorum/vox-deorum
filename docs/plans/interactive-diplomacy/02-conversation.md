# Stage 2 — vox-agents + Web: human↔LLM civ-to-civ conversation (CORE MVP)

> Part of the interactive-diplomacy plan. Shared design and watch-items live in [README.md](README.md); requirements in [specs.md](specs.md).

## Objective

A human seated as a major civilization can, **from the Web**, open a free-text conversation with an LLM-played civ and exchange messages; the LLM defaults to the voice of **that seat's configured diplomat agent** (specs §1, §7), with an explicit local-operator override available for debugging. The transcript persists through the stage-1 store and survives a restart. Closing the conversation writes a special message and **locks it for the rest of the current turn** (specs §8). No deals yet — this is the conversation MVP, the easiest direction (human→LLM) built end to end.

## Approach

Diplomacy is inherently civ-to-civ and **direction-agnostic** (specs §2, §5): a conversation is keyed by the player pair ordered by `playerID` plus the current speaker/recipient, either of which may be human or LLM. Today's chat API is "client names an agent"; for diplomacy the Web client names the target civ and the server defaults to that seat's configured diplomat while allowing a local debug override. To make the in-memory thread map 1:1 onto the stage-1 transcript rows, **restructure `EnvoyThread`'s identity to mirror the mcp-server's message shape** — the endpoint pair, each endpoint's `human`/`llm` role, and the current speaker — replacing the single `playerID` + free-form `userIdentity`. Do this as one coordinated refactor across `EnvoyThread`, chat routes, endpoint labeling, context/parameter resolution, and `getPlayerAssignments`, so diplomacy and ordinary envoy/telepathist chat share one endpoint-pair model. The in-memory `chatSessions` map stops being the source of truth and becomes a **write-through cache** over the stage-1 transcript store.

## Work items

1. **`vox-agents/src/types/chat.ts` (+ consumers)** — **restructure `EnvoyThread`'s identity to align with the mcp-server transcript shape** (stage-1 §1/§3): replace the single `playerID` + free-form `userIdentity` with an endpoint pair that maps 1:1 onto `append-message` / `read-transcript` for diplomacy — `playerAID` / `playerBID`, `playerARole` / `playerBRole` (`'human' | 'llm' | 'observer'`), and `speakerID` (specs §2, §6). A reserved `-1` marks the observer/no-seat side so ordinary envoy/telepathist chat still fits the same shape; durable transcript storage remains major-civ-only and should reject `-1`. Ordering to `Player1ID` / `Player2ID` stays server-side in the store tools, so the thread can keep endpoint-oriented A/B. Migrate the existing consumers off `playerID` / `userIdentity`: `envoy.ts` `formatUserDescription` derives its persona string from the non-speaker endpoint's civ + role instead of `userIdentity.role` / `displayName`; `web/routes/agent.ts` and `types/api.ts` populate the pair (mapping the context-derived seat to the `llm` endpoint and the caller to the human/observer endpoint); and the UI (`AgentSelectDialog.vue`, `ChatDetailView.vue`) selects and labels both endpoints by ID instead of the `'observer'` string sentinel.
2. **`vox-agents/src/types/config.ts`** — add optional `diplomat` and `negotiator` fields to `PlayerConfig` beside `strategist` (the `negotiator` is unused until stage 5 but declared now so seat config is complete). They select agents the same way `strategist` does, and may carry their own model overrides via the existing per-agent `llms` map.
3. **Endpoint resolution (initiator + target).** The Web client is a single-operator **debug surface**, so the operator may choose freely — any initiator seat, any target civ, and, if they want, an explicit voicing agent. When a conversation is opened against an LLM civ, **default** the voice to **that target seat's** configured `diplomat`, resolved from `getPlayerAssignments` / the session config (specs §2, §7); the operator can override it. The initiator endpoint is likewise whatever seat the operator selects, defaulting to the session's human-control seat when one exists. Extend the assignment shape as needed so it reports the resolved diplomat/negotiator names and models, not just strategist data. Resolve the diplomat's VoxContext/parameters from the target seat, not from the initiating seat, so the agent's `YouAre` metadata matches the civ being voiced. No server-trust enforcement is required at this stage — this is a local debug client.
4. **`vox-agents/src/web/routes/agent.ts`** — reuse `/api/agents/chat` and `/api/agents/message` for the diplomacy path. Add diplomacy-aware request fields / mode handling so `/api/agents/chat` opens or finds the one conversation for a player pair ordered by ID, hydrates it from `read-transcript`, and resolves the target seat's diplomat server-side as the default. Use a deterministic cache key for diplomacy chats, such as game ID + ordered pair, so repeated opens hydrate the same conversation rather than minting parallel threads. `/api/agents/message` appends the human text through archival `append-message`, executes the resolved target diplomat through the target seat's context/parameters, appends the LLM response, and streams the response over SSE from vox-agents. `append-message` itself never streams or triggers the response. The diplomacy path **defaults** the voice to the target seat's diplomat but honors an explicit `agentName` override from the debug client; existing named-agent chat behavior remains for non-diplomacy chats.
5. **`close-conversation` tool + semantics** — give the `Diplomat` envoy a **`close-conversation`** tool that writes the close special message via `append-message`, using the endpoint pair already on the thread. vox-agents derives **open/closed status (and the same-turn resume lock)** from the presence and turn of that message (specs §8). Because there is one conversation per pair, "reopen" means resuming the same transcript on a later turn; the counterpart cannot resume it on the turn it was closed. (The deal-handling tools `propose-deal` / `forward-deal` are added to the diplomat later, in stage 5.)
6. **Web close control** (`vox-agents/ui` — `ChatDetailView.vue` / the chat components) — a **Close** action on the conversation so the human can close it from the Web, and a read-only **open/closed indicator** reflecting the derived status (closed-this-turn vs. reopenable). Closing from the Web routes through the same `append-message` close path as the diplomat's tool. Reuse existing chat and state styles; the UI change is a compact conversation control, not a new landing surface.
7. **`vox-agents/src/infra/agent-registry.ts`** — ensure the `Diplomat` envoy is registered/selectable per seat (it already exists); no negotiator yet.

## Reuse

`Diplomat` / `LiveEnvoy` / `Envoy` (`src/envoy/*.ts`) and their existing tools (`get-briefing`, `get-diplomatic-events`); the chat routes + SSE streaming and `chatSessions` map (`src/web/routes/agent.ts`); the per-seat agent-selection idiom used for strategists; `getPlayerAssignments`; the human-control seat resolver; the Vue chat surface (`ui/src/components/chat/*`, `ChatDetailView.vue`) and `ui/src/api/client.ts`.

## Verify

End to end on the Web: launch a session with an LLM civ whose seat config names a diplomat. A human seated as another civ opens a conversation with it and exchanges free-text; the LLM replies in the configured diplomat's voice (and uses the configured model), unless the local operator selected an override. Stop and restart; reopen the conversation — the full transcript is still there (read from the store, not memory). Close the conversation; confirm it cannot be reopened on the same turn but can on a later turn.

## Done when

A human can hold a durable, restart-surviving, civ-to-civ free-text conversation with an LLM-played civ on the Web, voiced by the target seat's configured diplomat by default, with working close-and-lock semantics — and the transcript is the mcp-server store, with `chatSessions` reduced to a write-through cache.
