# Refactor diplomats to speak via a `send-message` tool

## Context

Today the live envoys (Diplomat, Spokesperson) speak to the counterpart by emitting **raw assistant free text**, while everything else (negotiation, briefings, closing) goes through tools. With `toolChoice: "auto"`, the model must constantly decide "do I write text now, or call a tool?" — and it gets this wrong (e.g. narrating instead of acting, or calling a tool when it should just reply). The free-text/tool duality is the root of the confusion.

This refactor makes **speaking a tool too**: a single `send-message` tool whose `Message` argument is streamed to the client exactly like free text is today. Once "speak" is an explicit action, the model's action space is uniform — it always picks a tool — which removes the ambiguity. Additionally, when a deal authored by the **counterpart** is open on the table, the Diplomat is restricted to just two tools (`call-negotiator`, `send-message`) so it can't wander into briefings/analyst calls when the ball is in its court.

Decisions confirmed with the user:
- **Scope: all live envoys.** Diplomat *and* Spokesperson speak only through `send-message`. The shared mechanics live on `LiveEnvoy`.
- **Terminal tools.** `send-message`, `call-negotiator`, and `close-conversation` are all terminal for the Diplomat (Spokesperson: just `send-message`). The model may call `send-message` alone, `call-negotiator` alone, or both in one step — any of these ends the turn.
- **Deal-gating applies only to a counterpart-authored open proposal.** When *our* side has a pending proposal the ball is on the other side, so no restriction.

This plan also bundles a **second, related workstream**: a *cache-aware envoy prompt structure*. The settled conversation (everything before the active exchange, hydrated from the durable store) is collapsed into a single, cacheable `# Past Conversation` user block that **includes deal summaries** (today the terms are stripped, leaving only each deal's one-line text); the current activity (reasoning/tool-call trail) trails it additively and is re-compacted only when a size threshold is crossed. This is reusable so the thread can be compacted dynamically. See "Cache-aware envoy prompt structure" below.

## Key constraint discovered

`vox-context.ts` (line ~773) rewrites `toolChoice: "required"` → `"auto"` for the **anthropic** provider. So we cannot hard-force tool calls on Anthropic models; the prompt must steer it, and we keep a **free-text fallback**: if the model emits raw text instead of `send-message`, that text is still streamed natively and archived as the reply. This makes the change safe regardless of provider.

## Streaming design (the careful part)

The AI SDK (v6.0.174) delivers `tool-input-start` / `tool-input-delta` / `tool-call` / `tool-result` chunks to `streamText`'s `onChunk`. We turn the `send-message` argument back into clean streamed text:

- The web route's `OnChunk` runs every chunk through a per-request **send-message streamer**:
  - `tool-input-start` (toolName `send-message`) → open a text stream keyed by `chunk.id`.
  - `tool-input-delta` for that id → accumulate raw JSON, decode the partial `Message` string value so far, emit the **new suffix** as a normal `text-delta` SSE event (`{ type: 'text-delta', text, id }`), tracking how much was already emitted.
  - `tool-call` (toolName `send-message`) → authoritative full input; emit any remaining suffix (covers providers that stream the arg whole, e.g. Gemini), then close the stream.
  - `tool-call` / `tool-result` for `send-message` are **swallowed** (not forwarded) so the UI renders the message as a text bubble, never a tool-call card.
  - All other chunks (native `text-delta` for greetings/fallback, other tools' `tool-call`/`tool-result`) pass through unchanged.
- **Partial-JSON decode is synchronous** via a small hand-rolled `decodeJsonStringField(raw, "Message")` that walks the JSON string honoring escapes and backs off at an incomplete escape/`\u` at the buffer boundary. We deliberately avoid the SDK's exported `parsePartialJson` because it is `async`, and `concurrency.ts` does not await `onChunk` — async work there would race chunk ordering. The `send-message` schema is a single `Message` string field so it is always first/unambiguous.
- The nested negotiator runs with **no callback** (agent-tools call `context.execute` without one), so its tokens never leak to the client — confirmed in `agent-tools.ts`.
- **Greeting/special mode also speaks through `send-message`** (unify everything): `LiveEnvoy.prepareStep` sets `activeTools = ["send-message"]` (not `[]`) in special mode, so the greeting is a `send-message` call streamed as `text-delta` like any other reply — one path for all spoken output.

## Cache-aware envoy prompt structure (second workstream)

Today `LiveEnvoy.getInitialMessages` rebuilds the whole conversation into many small `ModelMessage`s every run, deal rows are reduced to their bare one-line text (terms stripped), and there is no cache breakpoint — so the long stable history is neither compact nor cacheable. Restructure the envoy prompt so the settled history is one stable, cacheable block and only the current activity grows.

### Shape
```
[system]                                   (getSystem)
[game context messages]                    (buildGameContextMessages — stable)
{ role: 'user', content: "# Past Conversation\n<compacted transcript + deal summaries>",
  providerOptions: { anthropic: { cacheControl: { type: 'ephemeral' } } } }   ← CACHE BREAKPOINT
<live tail: the not-yet-compacted recent messages, emitted verbatim/additively>
{ role: 'user', content: hint + addon }     (turn-specific — uncached)
(Diplomat) { role: 'user', content: dealContext }   (current on-the-table deal — uncached)
```

### Reusable compaction (new module `vox-agents/src/envoy/utils/conversation-compaction.ts`)
- `renderPastConversation(headMessages, thread)` — pure renderer that walks the head in order and produces the single `# Past Conversation` string, grouped by turn:
  - `## Turn <N>` header whenever the turn metadata changes; speaker-labelled lines separated by `--`:
    - voiced seat (assistant) → `You: <text>`.
    - counterpart (user) → `<Role> of <Civ>: <text>` from the thread's stored audience role + identity (e.g. `Diplomat of France:`). Reuse the existing `roleOf` / `identityOf` (already used by `formatUserDescription`) so the label matches the rest of the prompt.
  - `deal` rows → render through the **same deal renderer the live on-the-table view uses** (see below) — identical formatting for live and compacted deals, no bespoke summarizer.
  - assistant verbose trails (reasoning + tool-call parts) → collapsed to their **spoken result**, so the head is always clean regardless of whether it came from hydrated store rows or in-memory trails. In particular, **`send-message` compacts from a tool call into a message**: in the live tail it is presented as a `send-message` tool call, but in the head it renders as a `You: <Message>` line. This matches the durable store, where `send-message` is archived as a plain `text` row — so a reload (re-hydration) yields the already-compacted message form. The renderer must therefore map **both** representations to the identical `You: <text>` line (the in-memory `send-message` tool-call `input.Message`, and the re-hydrated string `text` content), so the head is byte-stable across the reload boundary and stays cacheable. (The negotiator handoff already leaves a `deal` row, rendered separately above.)
- **Reuse, don't reinvent the deal format:** extract the single-deal block out of `formatDealContext` (`diplomat-utils.ts`) into a shared `formatDealBlock(deal, messageType, status, viewerID, civName, inspection?)` (terms via `formatDealTermsByDirection` + a status/header line + the one-line `message`). `formatDealContext` (the live current-deal view) calls it for the active proposal; the compaction renderer calls it for each historical `deal` row (omitting the live-only `inspection`/promise-agreeability and inward rationale). So a deal reads the same in `# Past Conversation` as it does on the table.
- `splitForCompaction(messages, boundary, threshold)` — returns `{ head, tail, boundary }`. The head is `messages[0..boundary]`; the tail is the rest, emitted verbatim. The boundary is **sticky**: it only advances (folding more into the head) when `countMessagesTokens(tail)` exceeds `COMPACT_THRESHOLD`, so between advances the head is byte-stable and the provider caches up to the breakpoint; crossing the threshold pays a single re-cache. Reuse `countMessagesTokens` (`utils/models/token-counter.js`).

### Boundary state
- The initial boundary = the **hydrated length**: `syncThreadMessages` (`utils/diplomacy/transcript.ts`) is the single hydration point — after it rebuilds `thread.messages`, record `boundary = thread.messages.length` so "everything that came before the active thread (hydrated)" is compacted from the first live turn. Stored in a module-level `Map<threadId, number>` (server-only; never on the wire-serialized `EnvoyThread`). Re-hydration resets it; the threshold advances it. Ordinary (non-diplomacy) chats have no hydration → boundary defaults to 0 and grows by threshold only.

### Integration
- **`vox-agents/src/envoy/live-envoy.ts`** `getInitialMessages` (normal mode only): replace the `convertToModelMessages(filterSpecialMessages(input.messages))` block with: `splitForCompaction` → push the `# Past Conversation` user message (with the cacheControl breakpoint) → push `convertToModelMessages(tail)` (existing per-message conversion still handles the live tail: turn prefixes, reasoning drop, and the `send-message` tool-result skip from the first workstream). Special/greeting mode keeps skipping history.
- The Diplomat's `getInitialMessages` override (deal context) and the hint/addon are unchanged — they append after the tail and stay uncached, which is correct (turn- and deal-state-specific).

## Changes by file

### New files
- **`vox-agents/src/envoy/utils/conversation-compaction.ts`** — `renderPastConversation`, `splitForCompaction`, and the per-thread boundary map (second workstream above; the deal block is the shared `formatDealBlock`, not a new summarizer).
- **`vox-agents/src/envoy/send-message-tool.ts`** — `createSendMessageTool(context)` via `createSimpleTool`. Schema `{ Message: z.string().describe(...) }`. `execute` just returns a short confirmation (e.g. `"Message delivered."`); it does not mutate the thread (streaming + archival are handled elsewhere). Export a `SEND_MESSAGE_TOOL = "send-message"` name constant.
- **`vox-agents/src/web/routes/send-message-stream.ts`** — `createSendMessageStreamer(emitTextDelta)` returning `{ handleChunk(chunk): boolean }` (true = swallowed), plus the pure, exported `decodeJsonStringField(raw, field)` for unit testing.

### Envoy core
- **`vox-agents/src/envoy/live-envoy.ts`**
  - `toolChoice = "required"` (neutralized to auto on Anthropic, but forces tools elsewhere).
  - `getActiveTools` base list adds `"send-message"`; `getExtraTools` registers `createSendMessageTool(context)` alongside `get-briefing`.
  - Add `protected getCompletionTools(): Set<string>` returning `{"send-message"}` by default.
  - Move the **shared stop logic** here (generalized from the current Diplomat): persist via `super.stopCheck` (Envoy), then — stop if any step called a completion tool; continue if the last step has a pending non-completion (support) tool; otherwise stop once any step produced spoken free text (the fallback). Preserves the diplomat's "work past the generic max-steps until you speak/act" semantics.
  - `prepareStep`: special mode sets `activeTools = ["send-message"]` (greetings speak via the tool too — see streaming section).
  - `getInitialMessages` (normal mode): emit the compacted `# Past Conversation` block + cacheControl breakpoint + verbatim live tail instead of converting every message (see the cache-aware workstream above).
- **`vox-agents/src/utils/diplomacy/transcript.ts`** — `syncThreadMessages`: after rebuilding `thread.messages`, set the compaction boundary for `thread.id` to the hydrated length (the single hydration point that seeds "everything before the active thread").
- **`vox-agents/src/envoy/envoy.ts`** — `convertToModelMessages`: in the tool-role branch, **skip `send-message` tool-result parts** (drop the "Message delivered." confirmation; if that empties the tool message, `continue` as today). Do **not** rewrite anything into text — `send-message` tool-call parts stay as tool-call parts, so the model's own history consistently shows it spoke by calling `send-message`. (Orphaned-tool-call tolerance is unchanged from today's behavior, where tool-results are already flattened away.)

### Agents
- **`vox-agents/src/envoy/diplomat.ts`**
  - `getActiveTools` adds `"send-message"`.
  - Override `getCompletionTools()` → `{"send-message","call-negotiator","close-conversation"}`. Delete the now-redundant `completionTools` static and the bespoke `stopCheck` (logic now shared in `LiveEnvoy`).
  - `prepareStep`: keep the non-diplomacy tool gating, and add (normal mode only): when the counterpart has an **open** proposal on the table, set `config.activeTools = ["call-negotiator","send-message"]`. The flag is computed once in `getInitialMessages` and read here via a module-level `WeakMap<EnvoyThread, boolean>` (avoids a second deal read per step and avoids leaking a transient field onto the wire-serialized `EnvoyThread`). Special mode never sets the flag, so it can't override the `["send-message"]` greeting tool set.
  - `getInitialMessages`: consume the new `buildDealContextMessage` return shape; push the deal-context text as today and store the counterpart-open flag in the WeakMap.
  - `getSystem`: rewrite the resources/expectations text — speak to the counterpart **only** by calling `send-message`; describe the negotiator handoff; when a counterpart deal is on the table, either hand it to the negotiator or reply with `send-message`. Update `getDefaultAddon` nudge to mention `send-message`.
- **`vox-agents/src/envoy/spokesperson.ts`** — `getActiveTools` adds `"send-message"`; system prompt updated to say it answers by calling `send-message` (replacing "do not send out in a text block" guidance).

### Deal context — unify diplomat & negotiator deal-state derivation
- **`vox-agents/src/utils/diplomacy/deal-reduce.ts`** — add two shared predicates so the diplomat and negotiator agree on deal state instead of re-deriving it inline:
  - `ownPendingProposal(reduction, agentSeat)` → `reduction.status === "open" && reduction.active?.SpeakerID === agentSeat`.
  - `counterpartOpenProposal(reduction, agentSeat)` → `reduction.status === "open" && !!reduction.active && reduction.active.SpeakerID !== agentSeat`.
- **`vox-agents/src/envoy/utils/diplomat-utils.ts`** — `buildDealContextMessage` switches from `readDealMessages` + `deriveActiveProposal` to **`readActiveProposal(player1ID, player2ID)`** (the exact helper the negotiator and accept route already use; returns the same `DealReduction` that `formatDealContext` consumes). Return `{ text?: string; counterpartHasOpenProposal: boolean }`, with the flag = `counterpartOpenProposal(reduction, thread.agent)`. Also extract the single-deal rendering from `formatDealContext` into the shared `formatDealBlock` (exported for the compaction renderer) so live and compacted deal blocks share one formatter.
- **`vox-agents/src/envoy/negotiator.ts`** — replace its inline `ownPending` / counterpart-proposal checks (around `getInitialMessages`) with the new `ownPendingProposal` / `counterpartOpenProposal` predicates, so both agents read deal state through one definition.

### Reply extraction / archival
- **`vox-agents/src/utils/diplomacy/transcript-utils.ts`** — add `collectSpokenReply(messages)`: join the `Message` inputs of assistant `send-message` tool-call parts in order; if none, fall back to `joinAssistantText` (raw free text). Keep `joinAssistantText` as-is. Re-export from `transcript.ts`.
- **`vox-agents/src/utils/diplomacy/chat-turn-commit.ts`** and **`vox-agents/src/web/routes/agent.ts`** (`respondToHumanDeal`) — replace the two `joinAssistantText(...)` reply-capture calls with `collectSpokenReply(...)`.

### Web route
- **`vox-agents/src/web/routes/agent.ts`** — in the `POST /agents/message` handler, build a `createSendMessageStreamer((text, id) => sendEvent('message', { type:'text-delta', text, id }))` and route `streamCallback.OnChunk` through it: `if (!streamer.handleChunk(chunk)) sendEvent('message', chunk)`. (The deal auto-response path has no client stream, so it only needs the `collectSpokenReply` archival change.)

### UI
- No change required: `send-message` is delivered as `text-delta` (already handled by `useThreadMessages.ts`), its tool chunks are swallowed server-side, and on reload the durable store returns clean `text` rows. (There are unrelated uncommitted edits in `useThreadMessages.ts` / `ChatDetailView.vue` — leave them alone.)
