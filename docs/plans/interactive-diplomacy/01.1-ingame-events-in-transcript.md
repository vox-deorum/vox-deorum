# In-game diplomacy in the transcript: deals, promises, war, friendship, denouncement

> Extends stage 1 (transcript store). The transcript becomes the complete bilateral record between two majors: chat *and* in-game diplomacy.

## Context

Deals negotiated through chat are recorded in the `DiplomaticMessages` transcript, but deals enacted in-game (AI↔AI, human↔AI on the trade screen) exist only as `DealMade` rows in `GameEvents` — diplomat agents and chat UIs never see them in conversation context. Promises made or broken in-game emit no events at all; they are internal `CvDiplomacyAI` state readable only through Lua getters. War declarations, friendships, and denouncements likewise never enter the conversation.

This stage bridges all of that into the per-pair transcript, and adds three new game events (`PromiseChanged`, `FriendshipDeclared`, `PlayerDenounced`) to the stream.

## Decisions

- In-game deals become transcript rows of the **existing `deal-enacted` type**, with `Payload.Source = 'game'` and no `ProposalMessageID`.
- Chat-enacted deals also fire `DealMade` in-game. The DLL marks them with a new **`EnactedByAgent`** payload flag (derived from `bTreatAsHumanToHuman`, which only the `deal:Enact` agent path ever sets — verified). mcp-server still stores the GameEvent (third parties keep seeing the deal) but **skips the transcript write** — the chat already wrote its own `deal-accept`/`deal-enacted` pair.
- `PromiseChanged` fires on real state transitions only: **Made / Ignored / Broken**. Not on clear-to-none (war's "promises voided" is annotated on the war transcript row instead), and not on expiry — expiry is lazy in the DLL (no setter runs; e.g. `GetNumTurnsExpansionPromise` just computes remaining turns), so a Made event instead carries a computed **`ExpiresOnTurn`**.
- Promises applied through chat deals (`Player:SetPromise`) fire the event with **`MadeViaAgentDeal = true`**: stored as a GameEvent, no transcript row (the chat deal record already lists them).
- Friendship (Declaration of Friendship) fires **only on non-deal paths**. DoF as a deal item — chat or in-game — is already part of the deal record, so those call sites don't fire.
- New transcript message types: **`declare-war`, `friendship`, `denounce`, `promise`**. They are bridge-only (refused by `append-message`), pair-visible, `Player1Role = Player2Role = "game"`.

## New events

**`DealMade`** (existing, one new field): `EnactedByAgent` (bool, optional) — true only on the agent chat path.

**`PromiseChanged`** (new; pair-private visibility):

| Field | Meaning |
|---|---|
| `PromiserPlayerID` | who made (or refused / violated) the promise |
| `RecipientPlayerID` | who it was made to (whose diplo AI tracks it) |
| `PromiseKind` | `MILITARY`, `EXPANSION`, `BORDER`, `NO_CONVERT`, `NO_DIGGING`, `SPY`, `BULLY_CITY_STATE`, `ATTACK_CITY_STATE` |
| `NewState` | `MADE` = agreed; `IGNORED` = refused the demand on the spot (never promised); `BROKEN` = made it, then violated it |
| `MadeOnTurn` | game turn of the transition |
| `ExpiresOnTurn` | for `MADE`: turn the binding window lapses (game-speed scaled); `-1` = binds indefinitely |
| `MadeViaAgentDeal` | true when applied through a chat deal |

`COOP_WAR` is excluded — it is a three-party commitment with its own state machine (`SetCoopWarState`), and its outcome already produces `DeclareWar`. A `CoopWarStateChanged` event is a possible follow-up.

**`FriendshipDeclared`** (new; met-players visibility): `{ PlayerID, FriendPlayerID }` — mutual DoF, fired once per declaration, non-deal paths only.

**`PlayerDenounced`** (new; met-players visibility): `{ DenouncingPlayerID, DenouncedPlayerID, Reason? }` — `Reason` is a short label mapped from the AI's denounce message (`IDEOLOGY`, `WARMONGER`, `SPIES`, …, else `GENERAL`).

**`DeclareWar`** (existing, unchanged): the bridge turns it into per-pair transcript rows annotated "voids all active promises between the two."

### mcp-server transcript bridge

New `src/knowledge/transcript-bridge.ts`: `bridgeGameEventToTranscript(store, id, type, payload)`, dispatching over the 5 event types. Rows are written via `storeTimedKnowledge('DiplomaticMessages', ...)` with `orderPlayerPair`, both-majors-only guard, `Player1Role = Player2Role = "game"`, `composeVisibility([a,b])`, and `Payload: { Source: 'game', EventID: id, ... }`. Player names / team membership come from `readPublicKnowledgeBatch("PlayerInformations", ...)` (same pattern as `get-diplomatic-events.ts:321-332`).

| Event | Skip when | Row type | Speaker | Content / payload |
|---|---|---|---|---|
| `DealMade` | `EnactedByAgent` | `deal-enacted` | FromPlayerID | Content via the existing `formatStringDeal` + `getDealExpirySuffix` over the pre-localized `FromGives`/`ToGives` extraPayload; `Payload.GameDeal = {FromPlayerID, ToPlayerID, StartTurn, PeaceTreatyType, FromGives, ToGives, TurnsRemaining}` |
| `DeclareWar` | originator non-major | `declare-war` | OriginatingPlayerID | one row per major member of `TargetTeamID`; "…declared war… This war voids all active promises between the two."; `Payload.PromisesCleared: true` |
| `FriendshipDeclared` | — | `friendship` | PlayerID | "A and B made a public Declaration of Friendship." |
| `PlayerDenounced` | — | `denounce` | DenouncingPlayerID | "A publicly denounced B." (+ reason) |
| `PromiseChanged` | `MadeViaAgentDeal` | `promise` | PromiserPlayerID | MADE: "A promised B: \<label>. (binding until turn N / binds indefinitely)"; IGNORED/BROKEN phrased accordingly; full event fields in Payload |

Game deals deliberately do **not** get a `Payload.Deal` (`DealPayloadSchema`): reverse-mapping numeric `ItemType`+`Data1-3` is lossy (cities, third-party terms, vassalage), and the absence of `Payload.Deal` is exactly what keeps `deal-reduce`, `DealMessageCard`, and `enact-agent-deal` idempotency untouched (all three verified — they key on `Payload.Deal` / `ProposalMessageID`, which game rows don't carry).

Promise labels reuse `PROMISE_METADATA` (`deal-metadata.ts`) plus a small local map for the four non-authorable kinds — `PROMISE_TYPES` itself is not widened (it gates what agents may author).

Hooking (`src/knowledge/store.ts`):
- `handleGameEvent`: after `setMetadata("lastID", ...)` (~276), call the bridge in a try/catch (log and continue — never block event storage). `data` already carries the merged extraPayload.
- `dropEventsAfterId` (~173): also purge bridged rows in the same queued operation — `DiplomaticMessages` where `json_extract(Payload,'$.EventID') >= id`. Chat rows have no `EventID`, so they are untouched; re-delivery after a DLL reconnect stays idempotent.

## Verification (live game)

1. Autoplay until two AIs trade → `read-transcript` for the pair shows the `deal-enacted` row with readable terms.
2. Chat-negotiate and enact a deal → `DealMade` arrives with `EnactedByAgent: true`; the transcript has only the chat-written `deal-accept`/`deal-enacted` pair — no duplicate.
3. Enact a chat deal containing a promise → `PromiseChanged` stored with `MadeViaAgentDeal: true`, no `promise` row.
4. Via FireTuner: `Player:SetPromise` on an AI pair → `promise` row with expiry annotation; `Team:DoDeclareWar` → `declare-war` rows; `Player:DoForceDoF` → `friendship` row; `Player:DoForceDenounce` → `denounce` row.
5. Web chat for an affected pair renders the new rows; the diplomat's context includes them; a DLL disconnect/reconnect produces no duplicate rows.

## Watch-items

- Multi-member declaring teams produce rows only for `OriginatingPlayerID` — acceptable, vox games run 1-player teams.
- `ForwardGameEvent`'s JSON buffer: new payloads are small; DealMade grows by one bool — no new risk.
