# Stage 7.03: civ5-dll + mcp-server + vox-agents: transport plumbing

> Part of the stage-7 sub-plan ([specs.md](specs.md); index [../07-ingame-panel.md](../07-ingame-panel.md)). No visible UI change: this stage makes the two directions of the pinned contracts real and proves them with probes, so stage 04 is pure wiring.

## Objective

Stand up the transport under the contracts pinned in specs: the four game events flowing game → mcp-server → vox-agents, the four push functions flowing back through a generic Lua-call passthrough, correct size behavior in both directions, and the read-only reflush/paging path: verified end to end with a temporary probe handler and no panel involvement.

Carry both observer flavors through this transport using the canonical effective-seat and `AsObserver` contract in [specs.md](specs.md).

## Work items

1. **`civ5-dll/CvGameCoreDLL_Expansion2/CvConnectionService.cpp`: buffer/pool fixes.** (a) In `BroadcastEventFromLua`, increase the 2 KB message document to 64 KB, optionally updating its twin allocations for symmetry: chat text and serialized deals overflow 2 KB today and ArduinoJson **silently drops members**. (b) In the Lua-call argument conversion path, increase the 64 KB argument document to 256 KB and check `overflowed()` after deserialization and serialization, failing the call with an explicit error instead of invoking Lua with dropped members. The capacity is a node pool, not a byte count, and nested transcript rows cost far more pool than wire. DLL rebuild; no save-format change.

2. **`mcp-server/src/tools/general/call-lua-function.ts`: the generic passthrough.** A thin tool `{ Name, Args[] }` → `bridgeManager.callLuaFunction(Name, Args)`, sibling of `lua-executor.ts`; register in `tools/index.ts`. This is the **only** new mcp-server surface for pushes: all push logic stays in vox-agents, and the mod owns the receiving functions (specs: transport stays transparent).

3. **`mcp-server/src/knowledge/schema/events/` + `mcp-server/src/server.ts`: event admission.** Four schemas modeled on `HumanDecision.ts` (numeric `PlayerID` required, optional literal-true `AsObserver`, `.passthrough()`): `DiplomacyPanelOpened.ts`, `DiplomacyChatMessage.ts`, `DiplomacyDealAction.ts`, `DiplomacyTranscriptRequest.ts`; register in the events index; add the four names to `eventsForNotification` in `server.ts`. Document `PlayerID` as the effective seat. It may exceed the major-civilization range and must not be capped. `AsObserver` follows [specs.md](specs.md): it is present as `true` only for the pure-observer flavor and omitted for normal players and human strategists. Without both schema registration and forwarding, events are dropped at the store gate or never forwarded.

4. **`civ5-mod/UI/VoxDeorumDiploPanel.lua`: push-function registration.** Register the four push functions (`VoxDeorumDiploBegin`/`Messages`/`Status`/`Delta`) via `Game.RegisterFunction`; each body fires the same-named `LuaEvents` and returns true. Add the panel-side row dedupe by transcript `ID`. (The functions land in the panel file now so stage 04's wiring is Lua-side trivial; a temporary probe listener printing payloads to `Lua.log` serves this stage's verify.)

5. **Paginated transcript read (shared extension).** No pagination exists anywhere today: the Web loads full history. Add optional `BeforeID` / `Limit` to the `read-transcript` tool and the `getDiplomaticMessages` getter (`mcp-server/src/tools/knowledge/read-transcript.ts`, `src/knowledge/getters/diplomatic-messages.ts`), returning the newest `Limit` rows (below `BeforeID` when paging) and enough to compute `hasMore`; add a `readTranscriptPage` wrapper beside `readTranscript` in `vox-agents/src/utils/diplomacy/transcript.ts`. Backward-compatible: no params = full history, existing callers unchanged.

6. **Live effective-seat attestation and observer limits.** Add a focused read-only mcp-server query, backed by game Lua, that returns the current active player, whether that player is an observer, the observer UI override, the derived effective seat, whether the state is pure observation, and whether the concrete slot is a native `CvDeal` participant. The authoritative game-side definition is TradeLogic's pre-mutation `VoxDeorumOpenDeal` range check: `0 <= playerID < GameDefines.MAX_MAJOR_CIVS`. The seat-state participant field must mirror that exact definition. Reuse the same derivation as `VoxDeorumSeat.lua`, and refresh it before each mutating in-game diplomacy event rather than relying on cached identity. Transcript append may admit a non-negative observer endpoint only when it equals the query's live pure-observer slot, has role `Observer`, and is paired with a valid major civilization. Keep `inspect-deal` and `enact-agent-deal` within their native participant limits: an out-of-range pure-observer slot has no live deal path. Keep the Web's `-1` sentinel behavior unchanged. Confirm that visibility is set only for valid major-civilization columns, while the real observer slot remains part of the ordered transcript pair.

7. **`vox-agents/src/envoy/ingame-bridge.ts`: the bridge skeleton.** A new module owned by `StrategistSession`, instantiated beside `humanDecisionBus` with new cases in the `mcpClient.onNotification` switch in `vox-agents/src/strategist/strategist-session.ts`. This stage implements:
   - **Two per-pair queues with separate ownership.** An action FIFO serializes mutating chat and deal events. A push FIFO orders every Lua call back to the game and also owns each complete read-only reflush task. The push FIFO continues while an action is active, so reopening can report `Begin.busy` and stage 04 can stream before `runChatTurn` finishes. Deduplicate inbound events by generated event ID because pipe and SSE delivery can overlap.
   - **Resolve and authorize the caller once at the event boundary.** Add a `resolveCaller(event, liveSeatState, counterpartContext)` helper that maps `{ PlayerID, AsObserver }` to `{ callerID, callerRole, callerIdentity, voiceOverride, isSeatHolder, canActOnDeals }`. Always use the event `PlayerID` as `callerID`, but require it and `AsObserver` to match the live seat-state query from item 6. For a valid pure observer, return role `Observer`, fixed identity `{ name: 'Observer' }`, voice override `spokesperson`, `isSeatHolder: false`, and `canActOnDeals` only when the concrete slot is a native `CvDeal` participant. For a normal player or human strategist, use the effective civilization's real `civIdentity`, leave the voice on the seated path, and set both capabilities true. Reject inconsistent or stale combinations with an error `Status` before any mutation. The fixed Observer identity and `civIdentity` are display inputs and never establish authority themselves. This keeps pinned observers indistinguishable from normal human players in the backend, as required by [specs.md](specs.md).
   - **`DiplomacyPanelOpened` bypasses the action FIFO:** immediately enqueue one atomic reflush task on the push FIFO. Inside that task, call `readTranscriptPage`, then send `Begin{hasEnvoy, busy, hasMore}` and all ordered `Messages{append}` batches before releasing the FIFO. Never call `openDiplomacyChat`, because it compacts and rehydrates the live cache. Pack batches to the approximate 32 KB wire budget. Read `hasEnvoy` as a property of the counterpart's live seat assignment, using the same computation for a diplomat or spokesperson caller, and read `busy` from item 8. Enqueuing the read and pushes as one task prevents a slower reflush from clearing rows pushed by a newer action.
   - **`DiplomacyTranscriptRequest` also bypasses the action FIFO:** enqueue one push-FIFO task that calls `readTranscriptPage(BeforeID)` and sends `Messages{prepend, hasMore}` before releasing the FIFO.
   - **`DiplomacyChatMessage` on the action FIFO:** use a temporary probe handler that appends through the transcript utilities and queues the committed row as `Messages{append}`. Mark it for deletion in stage 04. `runChatTurn` commits internally, so retaining the probe append would duplicate the message.
   - **Deal capability defense.** Admit `DiplomacyDealAction` only when fresh live-state resolution reports `canActOnDeals: true`. Do not reject a native-supported pure-observer event merely because `isSeatHolder` is false or `AsObserver` is true. An out-of-range, forged, mismatched, or stale observer slot fails before any deal path or mutation.
   - **Game-bound content conversion.** Strip the `!@#$%^!` delimiter and apply the already delivered, tested `markdownToCiv5` converter to all pushed `Content`. Durable transcript rows remain raw markdown.
   - **`DiplomacyDealAction` on the action FIFO:** queue an error `Status` ("not wired yet") through the push FIFO. Stage 04 replaces this stub.

8. **`vox-agents/src/utils/diplomacy/chat-turn-commit.ts`: export `isThreadBusy(threadId)`.** The in-flight set is private; the only current way to observe it is acquiring the lock. A read-only accessor feeds `Begin.busy` (shared extension: the Web could use it too).

## Reuse

The `HumanDecision` event pipeline end to end (schema → whitelist → `onNotification`: the model for item 3 and the switch cases); `bridgeManager.callLuaFunction` and the `ConvertJsonToLuaValue` structured-args path (the `presentHumanDecision` mechanism); `lua-executor.ts` as the tool template; `readTranscript`/transcript utils (`transcript.ts`) extended, not bypassed; `Game.RegisterFunction` (`CvLuaGame.cpp`, exposed to all game Lua).

## Verify

With the game, bridge-service, mcp-server, and a vox-agents interactive session running, and a prior **Web** conversation seeded for the pair:

1. Fire a synthetic `DiplomacyPanelOpened` from `lua-executor` → the probe prints `Begin` + `Messages` batches in `Lua.log` containing the Web-seeded rows (shared-storage proof), with correct `hasMore`; a synthetic `DiplomacyTranscriptRequest` pages older rows.
2. A `DiplomacyChatMessage` > 2 KB arrives intact at the bridge (broadcast bump verified); the same message appears via the probe echo and afterwards in the Web chat view (same store).
3. A >100-message transcript reflushes as multiple ordered `append` batches, each under the wire budget; an artificially oversized single push fails loudly with the `overflowed()` error, not silently.
4. Text containing `!@#$%^!` survives both directions sanitized (typed into the event via `lua-executor`; embedded in a seeded transcript row).
5. Duplicate delivery: re-emitting the same generated event ID is ignored. Two rapid `PanelOpened` events produce ordered atomic reflushes on the push FIFO, including while a chat event occupies the action FIFO. A final row committed during a reflush remains visible after the reflush completes.
6. Restart bridge-service mid-session. Confirm that the mod-registered push functions remain callable, or that the fallback re-registration hook fires. This resolves the open risk from the specs.
7. Fire all four event types with the live observer-slot `PlayerID` above the major-civilization range and `AsObserver: true`. Confirm every event crosses the schema, store, notification, and bridge pipeline without truncation or range rejection, caller resolution selects the Observer identity and spokesperson voice, and text rows append to and read back from the durable real-slot thread. Confirm only the valid major counterpart receives a transcript visibility flag. An arbitrary or stale non-major slot must still be rejected.
8. Fire a pure-observer event with the live concrete slot and `AsObserver: true`. If the slot is outside the native participant range, confirm caller resolution reports no deal capability and the action fails before the deal path or mutation. If a setup uses a native-supported observer slot, probe the same action path without treating that setup as a general guarantee. Repeat with `AsObserver` omitted, a forged slot, and stale live state, and confirm each invalid form fails before transcript or game mutation. Confirm the same counterpart-seat `hasEnvoy` result is used for normal, pinned-observer, and pure-observer opens.

## Done when

Every pinned contract, including the effective-seat and pure-observer fields, crosses the wire correctly in both directions under probe. Sizes and duplicates behave as specified, live-attested observer-slot IDs remain valid for transcript endpoints, out-of-range observer deal actions fail closed, forged or stale observer events fail closed, and the read-only reflush and paging path serves real transcripts. Stage 04 only needs to connect real handlers to the UI.
