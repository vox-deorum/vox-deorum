# Stage 7.03: civ5-dll + civ5-mod + mcp-server + vox-agents: transport plumbing

> Part of the stage-7 sub-plan ([specs.md](specs.md); index [../07-ingame-panel.md](../07-ingame-panel.md)). No visible UI change: this stage makes the two directions of the pinned contracts real and proves them with probes, so stage 04 is pure wiring.

## Objective

Stand up the transport under the contracts pinned in specs: the four game events flowing game → mcp-server → vox-agents, the four push functions flowing back through a generic Lua-call passthrough, correct size behavior in both directions, and the read-only reflush/paging path: verified end to end with a temporary probe handler and no panel involvement.

Carry both observer flavors through this transport using the canonical effective-seat and `AsObserver` contract in [specs.md](specs.md).

## Work items

1. **`civ5-dll/CvGameCoreDLL_Expansion2/CvConnectionService.cpp`: buffer/pool fixes.** (a) Outgoing broadcasts: raise the 2 KB message documents in `BroadcastEventFromLua` and its C++ twin `BroadcastEvent` to 64 KB: chat text and serialized deals overflow 2 KB today and ArduinoJson **silently drops members** on pool exhaustion. After building each document, check `overflowed()` and fail the broadcast loudly (error to the Lua caller, logged refusal) instead of sending a truncated event. The high-frequency `ForwardGameEvent` keeps its 2 KB document but gains the same `overflowed()` guard and log. (b) Server→game arguments: the 64 KB isolation document lives in `RouteMessage`, which re-serializes the incoming `args` and re-parses them before `HandleLuaCall` feeds `ConvertJsonToLuaValue`; raise it to 256 KB. On a `NoMemory` deserialization failure, `RouteMessage` currently logs, substitutes an empty argument array, and still invokes `HandleLuaCall`: change it to return an explicit `lua_response` error and never invoke the function with discarded arguments. Also stop ignoring the `serializeJson` return values on the same path. The capacity is a node pool, not a byte count, and nested transcript rows cost far more pool than wire. DLL rebuild; no save-format change.

2. **`mcp-server/src/tools/general/call-lua-function.ts`: the generic passthrough.** A thin tool `{ Name, Args[] }` → `bridgeManager.callLuaFunction(Name, Args)`, sibling of `lua-executor.ts`; export a `create*Tool` factory and add its one-line entry to `toolFactories` in `tools/index.ts`. `callLuaFunction` queues the call and fails fast with `DLL_DISCONNECTED` when the pipe is down; the tool surfaces that error to the caller verbatim. This is the **only** new mcp-server surface for pushes: all push logic stays in vox-agents, and the mod owns the receiving functions (specs: transport stays transparent).

3. **`mcp-server/src/knowledge/schema/events/` + `mcp-server/src/server.ts`: event admission.** Four schemas modeled on `HumanDecision.ts` (numeric `PlayerID` required, optional literal-true `AsObserver`): `DiplomacyPanelOpened.ts`, `DiplomacyChatMessage.ts`, `DiplomacyDealAction.ts`, `DiplomacyTranscriptRequest.ts`. Do **not** add `.passthrough()` to the schema literals: the store applies it centrally at validation time (`store.ts`). The events index is auto-generated: add the four files, then rerun `generate-index.js` instead of hand-editing `index.ts`. Add the four names to the `eventsForNotification` allowlist in `server.ts`. Document `PlayerID` as the effective seat. It may exceed the major-civilization range and must not be capped. `AsObserver` follows [specs.md](specs.md): it is present as `true` only for the pure-observer flavor and omitted for normal players and human strategists. Without both schema registration and forwarding, events are dropped at the store gate or never forwarded.

4. **`civ5-mod/UI/VoxDeorumDiploTransport.lua` (new): push-function registration and probe.** A new imported include (`import="1"` in `VoxDeorum.modinfo`; refresh with `update_md5.py`), included by the panel file above its mock include; stage 04 grows this same file into the transport driver when it swaps the mock out. It registers the four push functions (`VoxDeorumDiploBegin`/`Messages`/`Status`/`Delta`) via `Game.RegisterFunction` under the lazy-registration contract pinned in [specs.md](specs.md): a one-shot `EnsureRegistered()` runs inside the panel context on the first `LuaEvents.VoxDeorumDiploOpen` and before any diplomacy broadcast, never at context load. Each function body fires the same-named `LuaEvents` and returns true. Note that `lua-executor` scripts run in the ConnectionService's own Lua state and cannot reach this context's tables: probes trigger registration by opening the panel or by firing a handler inside the panel's context, not by calling `EnsureRegistered()` from the executor. The panel-side row dedupe by transcript `ID` already shipped in stage 01 (`m_rowByID` in `setRows`/`appendRow`/`prependRows`): verify it, don't re-add it. A temporary probe listener in this file prints push payloads to `Lua.log` for this stage's verify; stage 04 replaces it with the real driver bound to `VoxDeorumDiploUI`.

5. **Paginated transcript read (shared extension).** No pagination exists anywhere today: the Web loads full history. Add optional `BeforeID` / `Limit` to the `read-transcript` tool and the `getDiplomaticMessages` getter (`mcp-server/src/tools/knowledge/read-transcript.ts`, `src/knowledge/getters/diplomatic-messages.ts`). The getter is a Kysely SQL query ordered by `ID`: paging filters `ID` below `BeforeID` with a descending `.limit()` re-reversed to ascending order, fetching one extra row (or a cheap count) to compute `hasMore`. Apply `Limit` in SQL, before the getter's JS-side `speakerRole` filter, and return a continuation cursor from the raw scan (the lowest scanned `ID`, say `NextBeforeID`) with `hasMore` computed from the raw scan as well: a `Role`-filtered page may come back short or even empty, and without that cursor an empty filtered page with `hasMore: true` could never advance `BeforeID`. The panel path never combines `Role` with paging. Add a `readTranscriptPage` wrapper beside `readTranscript` in `vox-agents/src/utils/diplomacy/transcript.ts`. Backward-compatible: no params = full history, existing callers unchanged.

6. **Observer-slot transcript admission.** No seat-state query and no attestation: events originate from our own mod over the local pipe, and the bridge trusts the event-carried effective seat like every other game event (specs: caller trust). The effective seat is computed once, Lua-side, in the stage-01 shared helper and carried in the event. The only server-side work is storage tolerance: confirm (and where needed, relax) that `append-message` / `read-transcript` accept an out-of-range observer slot as one endpoint of the ordered pair, with role `Observer`, paired with a valid major civilization. Keep `inspect-deal` and `enact-agent-deal` within their native participant limits: an out-of-range slot's deal actions fail there naturally. Keep the Web's `-1` sentinel behavior unchanged. Confirm that visibility is set only for valid major-civilization columns, while the real observer slot remains part of the ordered transcript pair.

7. **`vox-agents/src/envoy/ingame-bridge.ts`: the bridge skeleton.** A new module owned by `StrategistSession`, instantiated beside `humanDecisionBus` with new cases in the `switch (params.event)` dispatch of `mcpClient.onNotification` in `vox-agents/src/strategist/strategist-session.ts`. This stage implements:
   - **Two per-pair queues with separate ownership.** An action FIFO serializes mutating chat and deal events. A push FIFO orders every Lua call back to the game and also owns each complete read-only reflush task. The push FIFO continues while an action is active, so reopening can report `Begin.busy` and stage 04 can stream before `runChatTurn` finishes. Deduplicate inbound events by generated event ID because pipe and SSE delivery can overlap.
   - **Resolve the caller once at the event boundary.** Add a small `resolveCaller(event, counterpartContext)` helper that validates shape — numeric `PlayerID ≠ CounterpartID`, counterpart seat context live — and maps `{ PlayerID, AsObserver }` to `{ callerID, callerRole, callerIdentity }`: the event `PlayerID` as `callerID` always; `callerRole: 'Observer'` with no civilization identity when `AsObserver` is true (display only), the effective civilization's real `civIdentity` otherwise. No live seat matching and no capability derivation: the voice is always the counterpart's assigned diplomat, for every flavor (specs: caller trust). An event failing the shape check gets an error `Status` and no mutation.
   - **`DiplomacyPanelOpened` bypasses the action FIFO:** immediately enqueue one atomic reflush task on the push FIFO. Inside that task, call `readTranscriptPage`, then send `Begin{hasEnvoy, busy, hasMore}` and all ordered `Messages{append}` batches before releasing the FIFO. Never call `openDiplomacyChat`, because it compacts and rehydrates the live cache. Pack batches to the approximate 32 KB wire budget. Read `hasEnvoy` as a property of the counterpart's live seat assignment, the same computation for every caller flavor, and read `busy` from item 8. Enqueuing the read and pushes as one task prevents a slower reflush from clearing rows pushed by a newer action.
   - **`DiplomacyTranscriptRequest` also bypasses the action FIFO:** enqueue one push-FIFO task that calls `readTranscriptPage(BeforeID)` and sends `Messages{prepend, hasMore}` before releasing the FIFO.
   - **`DiplomacyChatMessage` on the action FIFO:** use a temporary probe handler that appends through the transcript utilities and queues the committed row as `Messages{append}`. Mark it for deletion in stage 04. `runChatTurn` commits internally, so retaining the probe append would duplicate the message.
   - **Game-bound content conversion.** Strip the `!@#$%^!` delimiter and apply the already delivered, tested `markdownToCiv5` converter to all pushed `Content`. Durable transcript rows remain raw markdown.
   - **`DiplomacyDealAction` on the action FIFO:** queue an error `Status` ("not wired yet") through the push FIFO. Stage 04 replaces this stub.

8. **`vox-agents/src/utils/diplomacy/chat-turn-commit.ts`: export `isThreadBusy(threadId)`.** The in-flight set (`inFlight`) is module-private; the only current way to observe it is acquiring the lock. A read-only accessor feeds `Begin.busy` (shared extension: the Web could use it too).

## Reuse

The `HumanDecision` event pipeline end to end (schema → whitelist → `onNotification`: the model for item 3 and the switch cases); the auto-generated events index (`generate-index.js`); `bridgeManager.callLuaFunction` and the `ConvertJsonToLuaValue` structured-args path (the `presentHumanDecision` mechanism); `lua-executor.ts` as the tool template and the `toolFactories` registration pattern; `readTranscript`/transcript utils (`transcript.ts`) extended, not bypassed; `Game.RegisterFunction` (`CvLuaGame.cpp`, exposed to all game Lua; registry held DLL-side in `m_registeredFunctions`).

## Verify

With the game, bridge-service, mcp-server, and a vox-agents interactive session running, and a prior **Web** conversation seeded for the pair:

1. Open the panel once (or fire a transport handler inside the panel's context) so lazy registration runs, then fire a synthetic `DiplomacyPanelOpened` from `lua-executor` → the probe prints `Begin` + `Messages` batches in `Lua.log` containing the Web-seeded rows (shared-storage proof), with correct `hasMore`; a synthetic `DiplomacyTranscriptRequest` pages older rows. Confirm registration ran exactly once and nothing called into the connection service at context load.
2. A `DiplomacyChatMessage` > 2 KB arrives intact at the bridge (broadcast bump verified); the same message appears via the probe echo and afterwards in the Web chat view (same store). An artificially oversized broadcast exceeding the 64 KB pool is refused loudly by the new `overflowed()` guard rather than arriving truncated.
3. A >100-message transcript reflushes as multiple ordered `append` batches, each under the wire budget; an artificially oversized single push fails with the explicit `lua_response` error required by item 1, and the target function is never invoked with truncated or empty arguments.
4. Text containing `!@#$%^!` survives both directions sanitized (typed into the event via `lua-executor`; embedded in a seeded transcript row).
5. Duplicate delivery: re-emitting the same generated event ID is ignored. Two rapid `PanelOpened` events produce ordered atomic reflushes on the push FIFO, including while a chat event occupies the action FIFO. A final row committed during a reflush remains visible after the reflush completes.
6. Restart bridge-service mid-session and confirm the mod-registered push functions remain callable afterwards, per the DLL-side registry persistence pinned in [specs.md](specs.md).
7. Fire all four event types with an observer-slot `PlayerID` above the major-civilization range and `AsObserver: true`. Confirm every event crosses the schema, store, notification, and bridge pipeline without truncation or range rejection, caller resolution labels the caller `Observer` for display, and text rows append to and read back from the durable real-slot thread. Confirm only the valid major counterpart receives a transcript visibility flag, and that the same counterpart-seat `hasEnvoy` result is used for normal, pinned-observer, and pure-observer opens.

## Done when

Every pinned contract, including the effective-seat and pure-observer fields, crosses the wire correctly in both directions under probe. Sizes and duplicates behave as specified, observer-slot IDs remain valid transcript endpoints, and the read-only reflush and paging path serves real transcripts. Stage 04 only needs to connect real handlers to the UI.
