# Strategist Orchestrator — Staged Implementation Plan

This folder holds the strategist-orchestrator feature: the specification ([specs.md](specs.md)) and this staged implementation plan. Each numbered file is a self-contained stage — objective, approach, work items, what it reuses, and how to verify it. Implement in order: each stage is independently verifiable, and the `null` workflow runs end-to-end at Stage 4, so no large untested chunk is left for the end.

This document is the **build-order outline**. It fixes the stages, their order, and each stage's expectation at completion — not the per-stage detail. Individual stage plans are drafted later, one file each, in the house format (`## Objective` / `## Approach` / `## Work items` / `## Reuse` / `## Verify` / `## Done when`).

**Goal in one line:** a `scripted-strategist` runs a data-defined workflow through a real game exactly where a static strategist runs today, and an **offline orchestrator agent** reviews its recent runs and commits versioned edits to the workflow's script, manifests, and prompts — never touching the run in flight, never reaching mcp-server.

## Context

Today a strategist is TypeScript compiled into the app. The spec turns a strategist workflow into **editable data**: a small control-flow script, per-sub-agent manifests, and templated prompts, run inside a per-run working folder. It then adds a **non-blocking orchestrator** that reviews recent runs and iteratively rewrites those artifacts to grow a seed into plausibly good variants. The value is that workflows become improvable without recompiles, and an offline agent can debug and evolve them from real telemetry.

The spec is deliberately ambitious. This plan scopes a **maintainable v1**: it keeps the pieces that prove the whole pipeline end-to-end and **defers the machinery that is hard to change** until the loop has earned it. Deferrals were chosen with the maintainer and are listed explicitly; every one leaves a clean seam.

## Locked architecture decisions

These decisions shape the stages and are reflected in [specs.md](specs.md):

- **The orchestrator is a *type of agent*, like a strategist** — a registered agent with its own sub-agents and scoped file and git tools, driven offline by a console in the oracle/telepathist family. It is not a bespoke pipeline.
- **The decision-quality judge is one pre-defined sub-agent the orchestrator can call** — deferred from v1, but a cheap add once the orchestrator-as-agent machinery exists. No separate judging subsystem is ever built.
- **SES sandbox from the start.** The script substrate is Endo `ses` (lockdown plus capability Compartments that mirror the manifest allowlist model) inside a worker thread for the kill switch and memory cap. It runs at native V8 speed with ordinary `async`/`await`, so awaited and parallel sub-agent calls need no host-promise bridging, and it needs no native build step, which matters on Windows.
- **Git-backed versioning.** The orchestrator-managed workflow folder is a git repository the orchestrator has scoped access to. A commit is a version snapshot, the message is the changelog, `git log` is the history, and a per-seat branch or ref is the `current` pointer the runtime reads once at run start. Adoption runs the gate, then advances the seat ref atomically. There is no automatic rollback: `git revert` is a deliberate edit, matching the spec's stance.
- **One cost currency, cache-aware.** Cost-weighted tokens are the single unit, with per-tier and per-operation coefficients for cache reads, reads, and writes. Cached-token counts are captured where the provider reports them and **estimated from adjacent same-turn calls otherwise** — the estimate is in v1 so telemetry actually carries cache data to verify against.
- **One generic victory-trend interface, score provider first.** v1 ships the smoothed, phase-gated score ratio behind the interface; the external civ-bench connector implements the same interface later.

## Stages

Infra first, then the executable workflow, then signals, versioning, and the orchestrator.

Define cost-weighted tokens as the single unit with a per-tier and per-operation (cache-read, read, write) coefficient config, and surface cost per call. Verified against existing telemetry databases. |
| 2 | [02-working-folder.md](02-working-folder.md) | **Working-folder model and run-record renderer, online and offline.** The per-seat and per-run folder layout, and a single renderer that produces `state/*.json` snapshots and the markdown run record plus per-sub-agent transcripts from **either** the live knowledge reports (online, during a run) **or** the telemetry spans (offline, rebuilt for the orchestrator) — one code path, two sources. Recreate `null` and `none` and prove identical action tool-call sequences under a fixed seed through a real turn. The minimum thin slice. |

The allowlist design reserves a future memory/RAG tool grant. Verified with a single-LLM workflow, a budget-exhaustion test, and a parallel-fan-out test. |

The **adoption gate** is static checks over script, manifest, and prompt references, then a **stubbed dry-run** (stubbed models and actions) against a recent `state/` snapshot; on pass it advances the seat ref atomically. The dry-run harness doubles as the hand-authoring and development testbed. Verified by committing a hand-edited version that a live run pins and picks up mid-game. |
| 9 | [09-orchestrator-offline.md](09-orchestrator-offline.md) | **Offline orchestrator agent and console cycle.** The orchestrator as a registered agent with file-edit and scoped-git sub-agents and tools (and a reserved judge-sub-agent seam), driven by an offline console in the oracle/telepathist family; its working folder (workflow git, notes, recent run folders, signals, the tool and game-state schema snapshot from cached MCP definitions, the substrate reference, an example `state/`, and the tier list); a persistent conversation with stable prefixes and compaction; writing **only** drafts, commits, and notes; adopting **through the gate** (Stage 8). | 10 | [10-online-continuity.md](10-online-continuity.md) | **Cross-game continuity and online triggers (later phase).** The version line and notes follow the workflow across games (a new game's first cycle inherits prior history and notes through the same git line). **Online mode** fires a cycle during play on an in-game trigger (a cost spike against the rolling average; every N runs; a budget exhaustion or script failure; a victory-trend decline), config-gated, with the edit landing on a subsequent run and never the one in flight. | 11 | [11-boundary-review.md](11-boundary-review.md) | **Closing security and boundary review, and deferred-seam confirmation.** A static review that the orchestrator cannot reach mcp-server or issue a game action (only script, manifest, prompt, and note writes plus scoped git succeed); that the never-stall guarantees hold on every terminal path; and that committed versions are substantively distinct. Confirm each reserved seam stays cleanly addable: the judge sub-agent, the civ-bench connector, and `learned` plus RAG-backed memory. |

## What v1 defers (and the seam it leaves)

| Deferred | Why it is safe to defer | Seam left for it |
|---|---|---|
| Decision-quality LLM judge | The orchestrator judges from records, cost, coverage, and victory trend | The orchestrator is an agent (Stage 9); the judge is one more pre-defined sub-agent |
| civ-bench connector | The score ratio is a self-contained victory trend | The generic victory-trend interface (Stage 7); add a provider |
| `learned` strategist and episode retrieval | Baseline, briefed, and staffed already prove the substrate | The manifest allowlist design (Stage 5) reserves a memory/RAG tool grant |

The sandbox and cache-token estimation are **not** deferred: SES is adopted directly in Stage 3, and the adjacent-call estimate is folded into Stage 1 so telemetry has data to verify.

## How it fits the existing code

- **It plugs into the turn loop unchanged.** `scripted-strategist` registers in `vox-agents/src/infra/agent-registry.ts` and runs behind `vox-agents/src/strategist/vox-player.ts`'s pause, refresh, pace, decide, resume machinery, using `VoxContext`'s `withRun`/`forkRun` and token sink in `vox-agents/src/infra/vox-context.ts`. No new turn machinery.
- **The equivalence oracle already exists.** The static strategists' prompt builders in `vox-agents/src/strategist/agents/simple-strategist-base.ts` (`getSystem`/`getInitialMessages`) are canonical for the Stage 6 bootstrap tests.
- **The prompt template engine reuses `jsonToMarkdown`** in `vox-agents/src/utils/tools/json-to-markdown.ts` — the one converter for rendering game-state files into prompts.
- **`state/` snapshots reuse the knowledge reports** behind `get-players`, `get-cities`, `get-military-report`, `get-options`, `get-victory-progress`, and `get-events` in `mcp-server/src/tools/knowledge/`; `get-options` also defines the decision-coverage inventory. Offline, the same `state/` is rebuilt by replaying those tools' output spans.
- **Telemetry is extended, not replaced.** The SQLite span pipeline in `vox-agents/src/utils/telemetry/schema.ts` and the token capture in `vox-agents/src/infra/vox-context.ts` stay the source of truth; Stage 1 adds cached-token attributes plus the adjacent-call estimate, and Stage 2's renderer reads spans the way `vox-agents/src/oracle/retriever.ts` already does.
- **The orchestrator console mirrors oracle and telepathist** in `vox-agents/src/oracle/console.ts` (and telepathist's persistent, database-backed conversation) and reuses the cached MCP tool-definition snapshot the oracle already loads for the schema surface.
- **`shared/` replaces `workingMemory`.** The cross-turn area is the scripted counterpart of `StrategistParameters.workingMemory` (focus-briefer and episode requests) in `vox-agents/src/strategist/strategy-parameters.ts`.
- **Config resolution reuses `vox-agents/src/utils/config.ts`** (`configsDir`, `telemetryDir`), the `config.llms` model registry (tier-to-model), and per-player `PlayerConfig.llms` overrides.
- **mcp-server is essentially unchanged** — the runtime calls existing action tools (`set-flavors`, `set-research`, `keep-status-quo`, and the rest), and the schema snapshot builds on the existing cached tool-definition mechanism.

## Risks and watch-items (resolved inside the relevant stages)

- **Cache-estimate trust (Stage 1).** The adjacent-call estimate must be conservative and clearly flagged in telemetry against provider-reported counts, so budgets and the cost signal never silently trust a bad guess.
- **One renderer, two sources (Stage 2).** The online and offline paths must produce identical `state/*.json`, or equivalence and the orchestrator's evidence diverge; the shared code path is the guard.
- **SES capability surface (Stage 3).** Compartments expose only injected primitives, and the allowlist boundary is the trust boundary, so keep the injected surface narrow and audited.
- **Budgets drive graceful degradation (Stages 1 and 5).** Budgets are in cost units, so the currency must be trustworthy before budgets gate behavior; a runaway LLM loop and a `while (true)` must both die.
- **Version pinning against live commits (Stages 8 and 10).** A run reads its seat ref once at start; the atomic ref advance is what keeps an adopted edit off the run in flight.
- **The orchestrator boundary is enforced, not conventional (Stages 9 and 11).** Scoped git access must not become a path to mcp-server or a game action; the "cannot act" test is a first-class deliverable.

## Verification (high level; per-stage detail comes later)

- **Per stage:** mock-tier Vitest for the substrate, runtime, template engine, budgets, equivalence, gate static checks, and the boundary test; the stubbed dry-run (Stage 8) verifies control flow, rendering, and budget wiring without spending tokens.
- **End-to-end:** a `scripted-strategist` completing a full game with pacing, actions, telemetry, termination, and crash recovery unchanged (real and live tiers); and an offline orchestrator cycle committing a versioned, changelogged edit that a later run pins and adopts.
- **Acceptance for the workflow half:** every v1 roster member passes its one-time bootstrap equivalence test.
- **Acceptance for the orchestrator half:** a committed edit lands on a later run without blocking any strategist, its changelog cites the signals it read, and the boundary test proves no mcp-server and no action reach.
