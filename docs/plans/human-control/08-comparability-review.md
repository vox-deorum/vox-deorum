# Stage 8 — Static code review for comparability

> Part of the human-control plan. Shared design and watch-items live in [README.md](README.md); requirements in [specs.md](specs.md).

## Objective

Confirm by **static review of the code** that a human game lands in the analysis stack identically to an LLM game (spec §5). Live end-to-end game runs are done separately by a human and are not part of this plan.

## Review checklist

- **Same write path.** Human decisions go through the same action tools — and therefore the same `*Changes` knowledge tables, replay messages, and `VoxDeorumAction` events — as LLM decisions. No separate write path anywhere.
- **Decision cost.** Deliberation time is recorded where LLM token cost is recorded (turn-span attribute + accumulated per-player metadata), so the analysis stack sees a comparable per-decision cost field.
- **Pacing parity.** The human seat goes through the same `isScheduledDecision` / `shouldInterruptDecision` checks in `VoxPlayer` with no human-specific bypass — neither more nor fewer decision opportunities than an LLM under the same pacing config (spec §4).
- **Mixed-seat safety.** The decision bus is keyed by playerID and owned per session; LLM and stock-AI seats never touch it.
- **Crash recovery.** The recovery path cancels and re-requests a pending decision rather than dropping it; the panel re-receives the options event after a relaunch (spec §6).
- **Fairness by omission.** The panel renders only the human civ's own data; the observer UI is simply not loaded in human-control mode, so no other player's rationale can render (spec §3).

## Findings (static review, 2026-06-12)

All six items verified against merged code. **Result: PASS** — no separate write path, no pacing bypass, no fairness leak; mixed-seat isolation and crash recovery both hold. No fixes required. Two non-blocking notes recorded below.

- **Same write path — PASS.** `HumanStrategist.applyDecision` maps every submitted field onto the same MCP action tools an LLM issues, all via `context.callTool`: `keep-status-quo`, `set-flavors`, `set-research`, `set-policy`, `set-persona`, `set-relationship` ([human-strategist.ts:129-202](../../../vox-agents/src/strategist/agents/human-strategist.ts#L129-L202)) — entering the identical `tool.execute` path the LLM/null idiom uses ([vox-context.ts:251-272](../../../vox-agents/src/infra/vox-context.ts#L251-L272)). The `*Changes` writes, `VoxDeorumAction` LuaEvent, and `AddReplayMessage` replay line are all emitted *inside* the tools via the shared `pushAction` → `pushPlayerAction` ([player-actions.ts:30-42](../../../mcp-server/src/utils/lua/player-actions.ts#L30-L42)), not by the human path. The single rationale is computed once and replicated onto every call ([human-strategist.ts:135](../../../vox-agents/src/strategist/agents/human-strategist.ts#L135)). Grep of the human-control files for direct DB/event/Lua emission returned zero matches — no bypass exists. Legacy Strategy-mode `set-strategy` is intentionally omitted (Flavor-mode only, per spec §1).

- **Decision cost — PASS (note).** Deliberation time lands in the same two slots as LLM token cost: the per-turn `deliberation.ms` attribute is set on the *same* `turnSpan.setAttributes` call as `tokens.input/output` ([vox-player.ts:234-245](../../../vox-agents/src/strategist/vox-player.ts#L234-L245)), and the accumulated per-player `deliberationMs-<playerID>` metadata is written via the same `set-metadata` tool that writes `inputTokens-<playerID>` ([human-strategist.ts:225-241](../../../vox-agents/src/strategist/agents/human-strategist.ts#L225-L241), [vox-player.ts:288-293](../../../vox-agents/src/strategist/vox-player.ts#L288-L293)). The strategist stashes the per-turn value into `workingMemory["deliberationMs"]` for `VoxPlayer` to emit, because inside `getSystem()` the active span is `agent.<name>`, not the turn span — deliberate and correct. The value used is the panel-reported wall-clock `DeliberationMs` from the event payload, not a server recompute ([human-strategist.ts:208-212](../../../vox-agents/src/strategist/agents/human-strategist.ts#L208-L212)). *Note:* deliberation accumulates via read-add-write each decision (survives a mid-game relaunch), whereas tokens accumulate on the context and flush once at shutdown (reset to 0 on relaunch) — internally consistent with the token naming convention, and if anything more crash-resilient. Per the plan's open item, "comparable" means written to the same slots; nothing in this repo consumes either yet.

- **Pacing parity — PASS.** The decision gate `isScheduledDecision || shouldInterruptDecision` is a single block in the shared `VoxPlayer.execute` loop ([vox-player.ts:170-174](../../../vox-agents/src/strategist/vox-player.ts#L170-L174)); both gates are pure functions of turn / last-decision / pacing-config / state ([pacing.ts:57-68](../../../vox-agents/src/strategist/pacing.ts#L57-L68)) and take no strategist type. Grep across `strategist/` and the whole `pacing/` tree found no `if (human)` skip/force branch. `playerConfig.strategist` is only referenced for telemetry labels and for the `none`-engine path, all *after* the gate fires. The human strategist registers no extra opportunities and suppresses none — it runs only when the shared loop invokes it and returns `""` like `null-strategist`; `lastDecisionTurn` advances identically for all seats. A `HumanDecision` arriving with no pending request is a harmless no-op and cannot create an off-cadence decision.

- **Mixed-seat safety — PASS.** The bus is a `Map<number, PendingDecision>` keyed by playerID, an instance class with no module-level singleton ([human-decision-bus.ts:69-70](../../../vox-agents/src/strategist/human-decision-bus.ts#L69-L70)); `request`/`resolve`/`cancel` each operate only on the addressed playerID, so two seats cannot collide (unit test covers this). It is owned as a `readonly` field on `StrategistSession` ([strategist-session.ts:52](../../../vox-agents/src/strategist/strategist-session.ts#L52)), threaded into every `VoxPlayer` at construction, and resolved by the `HumanDecision` handler using the event's playerID ([strategist-session.ts:643-652](../../../vox-agents/src/strategist/strategist-session.ts#L643-L652)). Only `HumanStrategist` ever reads it ([human-strategist.ts:58-71](../../../vox-agents/src/strategist/agents/human-strategist.ts#L58-L71)); `simple-strategist*`, `null-strategist`, and `none-strategist` produce zero references — `VoxPlayer` populates `_humanDecisionBus` for every seat but non-human agents never read it. A stray event for an AI seat resolves nothing (no pending entry). The field is declared non-serialized beside `_pendingRefresh` ([strategy-parameters.ts:35-40](../../../vox-agents/src/strategist/strategy-parameters.ts#L35-L40)) and only ever reaches tools as `experimental_context`, never over the MCP wire.

- **Crash recovery — PASS.** On relaunch, `handleGameSwitched` calls `humanDecisionBus.cancelAll(...)` ([strategist-session.ts:451](../../../vox-agents/src/strategist/strategist-session.ts#L451)) and `shutdown` does likewise; `cancel` deletes the entry and **rejects** the pending promise ([human-decision-bus.ts:110-116](../../../vox-agents/src/strategist/human-decision-bus.ts#L110-L116)) rather than dropping it. The rejection unwinds through `HumanStrategist`'s `await`, is caught, and `VoxPlayer` still calls `resume-game` so the game is never left blocked. Recovery recreates each `VoxPlayer` and re-runs `execute()`, so the human strategist re-issues `present-decision`, which re-fires `VoxDeorumHumanDecision` and the panel re-receives the options event ([present-decision.ts:49-63](../../../mcp-server/src/tools/actions/present-decision.ts#L49-L63)). A late submission for the cancelled wait is a benign no-op. Belt-and-braces: `request()` itself supersedes any stale same-player entry. The observer-UI override is re-issued defensively in `recoverGame` ([strategist-session.ts:599-602](../../../vox-agents/src/strategist/strategist-session.ts#L599-L602)). *Awareness note:* re-presentation relies on the relaunch emitting `GameSwitched` with a fresh `gameID`; recovery always produces a new context, so this path is exercised — if a relaunch ever reused the gameID, only the supersede-on-next-request net would apply.

- **Fairness by omission — PASS (note).** The panel renders only the single-player report handed to it via `VoxDeorumHumanDecision(playerID, turn, options)` — six categories all sourced from `m_options`, and a leader row reading only `Players[m_playerID]` ([VoxDeorumHumanPanel.lua:1112-1209](../../../civ5-mod/UI/VoxDeorumHumanPanel.lua#L1112-L1209)). Other civs are touched only for public name/portrait plus the human's *own* stance/notes — no opponent rationale. The observer UI is the separate "AI Observer (JFD)" mod, injected into the mod list only when `aiObserverEnabled`, which is forced off for human control: `setAiObserver(!isInteractiveMode && !isHumanControl(config))` ([strategist-session.ts:148](../../../vox-agents/src/strategist/strategist-session.ts#L148), [vox-civilization.ts:140-154](../../../vox-agents/src/infra/vox-civilization.ts#L140-L154)). The panel/trigger never subscribe to `VoxDeorumAction`/`VoxDeorumPlayerInfo`. The top-panel pin (`Game.SetObserverUIOverridePlayer`, set before `SetAIAutoPlay`) is a backstop against auto-switching ([strategist-session.ts:488-494](../../../vox-agents/src/strategist/strategist-session.ts#L488-L494)). *Note:* the always-loaded `VoxDeorumTest.lua` does subscribe to `VoxDeorumAction`/`PlayerInfo`, but it only `print()`s to Lua.log and forwards bridge events — it renders nothing on screen, so it is not a leak. `AddReplayMessage` writes to the acting player's own replay log (the §5 comparability path), not a cross-player render surface.

## Done when

Every checklist item is verified against the merged code, with findings (and any fixes) written down. The remaining validation — full human games, mixed human+LLM games — is run by a human per the success criteria in [specs.md](specs.md).
