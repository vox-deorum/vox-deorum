# Human Control — Staged Implementation Plan

This folder holds the human-control condition: the specification ([specs.md](specs.md)) and this staged implementation plan. Each numbered file is a self-contained stage: objective, work items, what it reuses, and how to verify it. Implement in order — each stage is independently verifiable, and the panel stages (5–7) are gated by the approved mockup (stage 4).

The goal in one line: a person occupies a strategist seat — a `human-strategist` registered in the agent registry and assignable in `llmPlayers` like any other — and steers one civ through the *same* influence-level action space the LLM strategists use, landing in the same telemetry, replay, and `game_outcomes` databases so human and LLM games are analyzed by the same tools.

Stages 1–3 build the full pipeline with the panel **simulated** via the `lua-executor` MCP tool (injecting a synthetic `Game.BroadcastEvent("HumanDecision", ...)`). Stages 5–7 replace the simulation with the real, incrementally built in-game panel.

## How it fits the existing code

- **`human-strategist` follows the `null-strategist` idiom.** `NullStrategist` (`vox-agents/src/strategist/agents/null-strategist.ts`) does all its work inside `getSystem()` via `context.callTool(...)` and then returns an empty string, which `VoxContext` treats as "no model call" — the LLM loop is skipped entirely. The human-strategist does the same: present the decision, block on the human, map the submission onto action tools, return `""`.
- **The panel's options are fetched server-side by `present-decision`, not handed over by the strategist.** The `human-strategist` triggers `present-decision` with just `{PlayerID, Turn}`; that tool calls `get-options` (Flavor mode) itself and pushes the strongly-typed `OptionsReport` into the panel. This avoids round-tripping the report out of the server and back in as a tool argument (which would also strip its type). Since `VoxPlayer.execute` already paused the game and `get-options` reads cached knowledge, `present-decision`'s fetch returns the same snapshot the strategist's own `ensureGameState`/`refreshGameState` cached for the turn — the panel and the strategist's action-tool mapping see one consistent option landscape.
- **The game is already paused across the wait.** `VoxPlayer.execute` calls `pause-game` before, and `resume-game` after, the strategist runs. An unbounded `await` inside `getSystem` simply holds that pause; no new pause machinery.
- **The decision bridge** (the one genuinely new mechanism): a **per-session** decision bus — a map from playerID to a pending `{resolve, reject, requestedAt}` — **owned by the `StrategistSession` instance**, not a module-level global, so one vox-agents process could in principle run multiple games each with its own bus. The session threads its bus to each `VoxPlayer` (constructor argument) and on to the strategist via a non-serialized `parameters._humanDecisionBus` field (mirroring the existing `_pendingRefresh` internal-field convention in `strategy-parameters.ts`). The strategist awaits `bus.request(playerID)`; the session's notification handler resolves it when the `HumanDecision` event arrives.
- **The inbound decision path reuses the notification channel.** `Game.BroadcastEvent` → DLL → bridge → the MCP store's `handleGameEvent` → `sendNotification` → the vox-agents MCP client → the `StrategistSession` notification switch. `HumanDecision` is schema-registered because the store drops unknown event types before notification routing. The notification keeps canonical routing metadata (`event`, `playerID`, `turn`, `latestID`) at top level and carries the validated event payload inside a top-level `data` object, so the human's choices ride the notification without being spread over routing fields. The same payload-forwarding path applies to every whitelisted event, not only to `HumanDecision`.
- **The outbound "present decision" path reuses the `LuaFunction` pattern** from `mcp-server/src/utils/lua/player-actions.ts`: the `present-decision` tool fetches the turn's options via `get-options` and hands them to a preregistered Lua function over the bridge batch queue (never inline-interpolated into a script), firing `LuaEvents.VoxDeorumHumanDecision(...)` into the panel. The `OptionsReport` is passed as a structured argument: the bridge JSON-serializes it for transport and the DLL's `ConvertJsonToLuaValue` rebuilds it as a native Lua table, so the panel reads its fields directly (stage 6) rather than parsing JSON in Lua.
- **mcp-server additions stay localized.** Stage 2 adds the `HumanDecision` event schema and notification whitelist entry, generic payload forwarding under `data`, the LuaFunction-backed `present-decision` tool, and a regular server-side heartbeat timer so the SSE channel survives an unbounded human pause. Everything else in the analysis and action pipeline is reused unchanged.

## Risks / watch-items

- **SSE liveness across an unbounded human pause.** Addressed by the always-on server-side heartbeat timer (stage 2); the interval must stay comfortably under the MCP client's 600-second body timeout.
- **Native-UI hijack scope (stage 6).** Verified in code: at autoplay activation the game makes the **observer slot the active player** (`CvGame.cpp`, `setActivePlayer(iObserver, ...)`); the native tech/policy screens key everything off `Game.GetActivePlayer()` (so as-is they'd render the observer's empty state, not the human civ's); and they commit choices via direct `Network.SendResearch` / `Network.SendUpdatePolicies` with no event hook to capture a selection in passing. A "hijack" therefore means **forking the screen Lua** (the EUI variants under `UI_bc1/`, since launches load Vox Populi + EUI) into `civ5-mod` as a re-pointed, commit-replaced chooser — see stage 6 for the reframed spike. Display-name suffixes are a non-issue either way: `set-policy` strips parenthetical suffixes (`" (New Branch)"` / `" (Policy)"`) server-side.
- **In-game takeover hotkeys.** Pressing `CONTROL_TOGGLE_AI_TAKEOVER` while the override is set ends autoplay, hands the participant *direct control* of the steered civ, and clears the override (`CvGame.cpp`); `CONTROL_TOGGLE_OBSERVER_MODE` is inert while the override is set. Covered by the spec's good-faith stance and supervised sessions — mention in the participant briefing; no code change.
- **Schema gate drops un-modeled events.** Keep the `HumanDecision` schema permissive (required `PlayerID` and `Rationale` only) so the panel can evolve its payload without silent drops at validation.
- **`SetObserverUIOverridePlayer` ordering and seating.** Use the seating-mapped actual playerID and set it before `SetAIAutoPlay` — the visibility copy happens at autoplay activation. Re-issue after crash recovery (defensive; the override is serialized in saves). The DLL side is "verify, not build."
- **Loading foreign saves.** Because the visibility copy happens only at autoplay activation, human-control games must start fresh (`gameMode: 'start'`) or resume saves created by a human-control session. Loading a save where autoplay began without the override will not retroactively copy visibility. Crash recovery of a human game is unaffected.
- **Decision bus stays per-session and keyed by playerID** so mixed seats and (in principle) concurrent games stay isolated.
